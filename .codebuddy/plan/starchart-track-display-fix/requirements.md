# 需求文档：StarChart Track 显示修复

## 引言

StarChart UI 的 Track 区域（武器轨道）目前存在两个视觉与交互问题：

1. **Track 上的部件显示不正确**：已装备到 Track 上的异形部件（如 1×2H、L 形等）没有以完整的形状和统一颜色展示，而是每个格子独立着色，无法直观看出部件的整体轮廓，且颜色与背包中显示的不一致。目标效果参考 HTML 原型（`Tools/StarChartUIPrototype.html`）中的 overlay 覆盖层渲染方式。

2. **拖拽放置时锚点不智能**：将部件拖到 Track 格子上时，系统以"鼠标悬停格"作为锚点（形状左上角起点），导致悬停在非左上角格时形状越界或位置错误。玩家期望：鼠标悬停的格子可以是形状中的**任意一格**，系统应反向推算出合法锚点，而不是死板地以悬停格为锚点。

> **可扩展性原则**：Track 的格子数量未来可能扩充到 5、6、7、8 格甚至更多（包括奇数格），形状种类也可能增加（如不同旋转方向的 L 形）。所有算法和渲染逻辑必须基于形状的**偏移列表**动态计算，严禁对 Grid 尺寸或形状种类做任何硬编码假设。

---

## 需求

### 需求 1：Track 上已装备部件以完整形状覆盖层展示，颜色与背包一致

**用户故事：** 作为玩家，我希望在 Track 上看到已装备部件的完整形状和颜色，与背包中看到的外观保持一致，以便直观了解部件占用了哪些格子。

#### 验收标准

1. WHEN 一个异形部件被装备到 Track 上 THEN 系统 SHALL 根据该部件的**偏移列表**动态计算 bounding box，并在该范围内渲染一个统一的覆盖层（overlay），覆盖所有被占用的格子。此逻辑对任意形状（包括未来新增的形状枚举）均适用，无需修改渲染代码。
2. WHEN 覆盖层渲染时 THEN 系统 SHALL 使用该部件类型对应的主题颜色（`StarChartTheme.GetTypeColor`）填充覆盖层背景，与背包中的颜色完全一致（Core 蓝色、Prism 紫色、LightSail 绿色、Satellite 黄色）。
3. WHEN 覆盖层渲染时 THEN 系统 SHALL 在覆盖层内部按实际形状（非矩形 bounding box）显示各格子的填充状态（occupied 格子显示颜色，empty 格子透明）。
4. WHEN 覆盖层渲染时 THEN 系统 SHALL 在覆盖层的锚点格（anchor cell）居中显示部件图标。
5. WHEN 覆盖层渲染时 THEN 系统 SHALL 对覆盖层添加类型颜色的边框或光晕，与 HTML 原型视觉风格一致。
6. WHEN 玩家 hover 覆盖层时 THEN 系统 SHALL 触发 tooltip 显示（与现有 `OnCellPointerEntered` 事件保持兼容）。
7. WHEN 玩家点击覆盖层时 THEN 系统 SHALL 触发 unequip 或 inspect 操作（与现有 `OnCellClicked` 事件保持兼容）。
8. IF 部件形状为 Shape1x1 THEN 系统 SHALL 退化为单格显示，行为与现有 `SetOverlay` 一致。

### 需求 2：拖拽放置时自动推算合法锚点

**用户故事：** 作为玩家，我希望将部件拖到 Track 的任意格子上时，系统能自动将形状放置到包含该格子的合法位置，以便我不需要精确瞄准锚点格也能成功放置。

#### 核心语义

形状是**固定不变**的。`Shape1x2H` 永远是横向两格，`ShapeLMirror` 永远是左上+左下+右下，形状本身不会因为拖到哪个格子而改变。

鼠标悬停的格子可以是形状中的**任意一格**，系统通过反向枚举候选锚点来推算合法放置位置：

```
对形状的每个 offset (dx, dy)：
    候选锚点 = (hoverCol - dx, hoverRow - dy)
    验证该锚点下形状所有格子都在 Track 边界内（0 ≤ col < GRID_COLS，0 ≤ row < GRID_ROWS）
    若合法 → 加入候选集

从候选集中选 row 最小、再 col 最小的那个
若候选集为空 → 标记为 Invalid（红色预览）
```

**此算法对任意 Grid 尺寸（2×2、3×2、4×2、奇数格等）和任意形状完全通用**，只需传入正确的 `GRID_COLS`、`GRID_ROWS` 和形状偏移列表，无需为特定尺寸或形状编写特殊分支。

#### 验收标准

1. WHEN 玩家将 Shape1x2H 部件拖到 2×2 Track 的右上角格（col=1, row=0）时 THEN 系统 SHALL 将该格视为形状的第2格（offset=1,0），反推锚点为 col=0，使形状占满上方两格（col=0,row=0 和 col=1,row=0）。
2. WHEN 玩家将 Shape1x2H 部件拖到 2×2 Track 的左上角格（col=0, row=0）时 THEN 系统 SHALL 将该格视为形状的第1格（offset=0,0），锚点为 col=0，形状占满上方两格。
3. WHEN 玩家将 Shape2x1V 部件拖到 2×2 Track 的左下角格（col=0, row=1）时 THEN 系统 SHALL 将该格视为形状的第2格（offset=0,1），反推锚点为 row=0，使形状占满左列两格（col=0,row=0 和 col=0,row=1）。
4. WHEN 悬停格对应的所有候选锚点均导致形状越界 THEN 系统 SHALL 将该放置标记为 Invalid（红色预览）。
5. WHEN 系统计算出合法锚点 THEN 系统 SHALL 同步更新拖拽预览高亮，使高亮格子与实际放置位置一致。
6. WHEN 计算出的放置位置与其他已装备部件重叠 THEN 系统 SHALL 按现有 Replace 逻辑处理（橙色预览 + 放置时顶出）。
7. WHEN Track 的 Grid 尺寸发生变化（如从 2×2 扩充到 3×2 或更大）THEN 系统 SHALL 无需修改锚点推算逻辑，仅更新 `GRID_COLS` / `GRID_ROWS` 常量即可正确工作。
8. WHEN 新增形状枚举（如 `ShapeLMirror` 或其他旋转变体）THEN 系统 SHALL 无需修改锚点推算逻辑，仅在 `ItemShapeHelper` 中添加对应的偏移列表即可正确工作。

#### 各形状行为验证（2×2 Track，GRID_COLS=2, GRID_ROWS=2）

| 形状 | 悬停左上(0,0) | 悬停右上(1,0) | 悬停左下(0,1) | 悬停右下(1,1) |
|------|-------------|-------------|-------------|-------------|
| `Shape1x2H` | 锚(0,0)→占上两格 ✅ | 锚(0,0)→占上两格 ✅ | 锚(0,1)→占下两格 ✅ | 锚(0,1)→占下两格 ✅ |
| `Shape2x1V` | 锚(0,0)→占左列两格 ✅ | 锚(1,0)→占右列两格 ✅ | 锚(0,0)→占左列两格 ✅ | 锚(1,0)→占右列两格 ✅ |
| `ShapeL` | 锚(0,0)→占左上+右上+左下 ✅ | 锚(0,0)→占左上+右上+左下 ✅ | 锚(0,0)→占左上+右上+左下 ✅ | **无合法锚点 → Invalid** ✅ |
| `ShapeLMirror` | **无合法锚点 → Invalid** ✅ | 锚(0,0)→占左上+左下+右下 ✅ | 锚(0,0)→占左上+左下+右下 ✅ | 锚(0,0)→占左上+左下+右下 ✅ |
| `Shape2x2` | 锚(0,0)→占全部4格 ✅ | 锚(0,0)→占全部4格 ✅ | 锚(0,0)→占全部4格 ✅ | 锚(0,0)→占全部4格 ✅ |

> **Invalid 是正确行为**：当悬停格不属于该形状能覆盖的任何合法位置时，放置应被拒绝。例如 `ShapeL`（缺右下）悬停右下角、`ShapeLMirror`（缺右上）悬停左上角，均无合法锚点。

## 附录：形状定义

所有形状偏移均相对于锚点 `(col=0, row=0)`，坐标系为 col 向右增大、row 向下增大。

| 形状枚举 | 偏移列表 | 视觉示意（■=occupied，□=empty） |
|---------|---------|-------------------------------|
| `Shape1x1` | `(0,0)` | `■` |
| `Shape1x2H` | `(0,0), (1,0)` | `■ ■` |
| `Shape2x1V` | `(0,0), (0,1)` | `■` / `■`（竖向两格） |
| `ShapeL` | `(0,0), (1,0), (0,1)` | `■ ■` / `■ □`（缺右下） |
| `ShapeLMirror` | `(0,0), (0,1), (1,1)` | `■ □` / `■ ■`（缺右上） |
| `Shape2x2` | `(0,0), (1,0), (0,1), (1,1)` | `■ ■` / `■ ■` |

> **关于 ShapeLMirror**：`ShapeL` 是左上+右上+左下（缺右下），`ShapeLMirror` 是左上+左下+右下（缺右上）。两者是同一 L 形的不同旋转方向，在代码中作为独立枚举值处理，各自有固定的偏移列表，不支持运行时旋转。

> **关于未来扩展**：新增任何形状只需在 `ItemShapeHelper.GetCells` 中添加对应的 `case` 分支和偏移列表，锚点推算算法、覆盖层渲染逻辑均无需修改。
