## 0. 为什么需要 Core Loop 文档（Why）

✅ **已锁定**

项目已经积累了大量内容：星图系统、敌人 AI、关卡模块、示巴星教学流、世界圣经。但**"玩家在任意一个随机 60 秒里到底在做什么"** 这个问题，目前还散落在各个子文档里，没有一个统一答案。

没有 Core Loop 共识会造成：

- 新功能设计无参照系——"这个功能要不要做？" 没有一根尺子衡量
- 美术/音效/UX 缺乏"要放大什么情绪"的锚点
- 排期无法判断优先级——所有功能看起来都"挺重要"
- 测试时没有"这次验收到底在验什么"的判断标准

Core Loop 文档不是给玩家看的，是给**团队做决策时看的**。它回答三个问题：

1. **60 秒循环**：玩家此刻在做什么？（行为层）
2. **三根支柱**：他为什么持续做下去？（动机层）
3. **MVP 证明**：做到什么程度，我们就知道这个循环是成立的？（验收层）

## 1. 60 秒循环（The 60-Second Loop）

> 这是 Core Loop 的核心——回答"玩家任意抽取的 60 秒里在干什么"。
> 如果无法用一张图和一段话讲清楚这 60 秒，说明核心体验还没有被定义清楚。

### 1.1 60 秒循环的主轴 ✅ **已锁定：探索主轴 — "听见，再进入"（Listen, then Enter）**

Project Ark 同时融合了**探索（银河城）**、**战斗（Twin-stick 射击）**、**构筑（星图）** 三大玩法。v1.0 定稿选定 **探索为主轴**，战斗作为"压力脉冲"嵌入，构筑作为"跨 Loop 余味"留在掩体/检查点处理。

> 60 秒 = 一次**房间进入 → 语义读取 → 压力应对 → 退出**的完整微旅程。

**标准 60 秒节拍**：

```
[0s]   推开门 → 新房间扑面而来：破碎的音叉树林 + 远处低频蜂鸣
[5s]   停下脚步，扫视环境 → 注意到墙上一行克拉德尼文 → 记下
[10s]  靠近音叉树 → 试射一发 → 发出 "Re" 音 → 周围冰雕共鸣
[20s]  触发共鸣机关 → 通路打开，但同时释放休眠爬行者 → 进入小规模战斗
[40s]  清场 → 拾取部件 → 读碎片文本 "它们只是在找同伴的声音"
[55s]  站在出口门前 → 犹豫要不要继续 → 最后一眼环视 → 推门
[60s]  下个房间，开始下一个 Loop
```

**这个游戏是**：一个"在大静默中当考古学家"的银河城，战斗是环境的一部分，不是核心。

**锁定这根主轴的三条硬理由**：

1. **与 World Bible 的情感基调一致**：第 0.3 节"庄严的孤独 + 易碎的共情"只能在玩家有时间"停下来听"时成立；持续战斗会淹没这两种情感。
2. **"理解即武器"只能在探索节奏里生效**：Design Pillar I 要求玩家通过"读懂世界"获得成长感，Twin-stick 肉鸽节奏不给这个时间窗。
3. **禁忌清单已经暗示过选择**：World Bible 禁止"小地图/任务标记/头顶血条"等即时反馈——这是探索型游戏的典型约束，战斗主轴下这些禁忌会变成"为难自己"。

**选择这根主轴带来的长期代价（必须承认）**：

- 对美术和关卡量级要求最高——每个房间必须"值得被探索"，不能靠重复敌人填充
- 当前完成度重战斗、轻探索——这是接下来 Feature 排期的**方向指针**
- 前 2 小时节奏会被 Twin-stick 习惯玩家评价为"慢"——示巴星教学流必须显式解决这个预期落差

**被 v1.0 拒绝的两条候选（历史存档，见附录 C）**：

- ❌ 候选 A：战斗主轴"读懂，再开火"——会让游戏被归类为"又一个 Twin-stick Roguelite"
- ❌ 候选 C：构筑主轴"采集，再编织"——60 秒内完成"采集→编织→验证"对 UX 要求过高，且星图面板当前是全屏大界面，与节奏不匹配

> ⚠️ 任何未来提案若违背"探索为主轴、战斗是脉冲、构筑是余味"的基本结构，应提出 `CoreLoop.md v2.0 Draft` 而不是在 v1.0 上打补丁。

### 1.2 60 秒循环的节拍图 ✅ **已锁定（基于探索主轴）**

```
        张力 (Tension)
         ▲
  高  10 │                ┌────┐
         │               ╱      ╲
   中 5  │      ┌────┐  ╱        ╲
         │     ╱      ╲╱          ╲
   低 1  │ ───╯                    ╲___
         └──────────────────────────────→  时间
         0s    10s   20s   30s   40s   60s

          进入   扫视   触发   战斗   清场   过门
          房间   线索   机关   脉冲   拾取   离开
```

**关键节拍说明**：

| 节拍 | 时长 | 玩家行为 | 情感 | 关键系统 |
|------|------|----------|------|----------|
| **进入** | 0-3s | 推门、停顿、第一眼 | 敬畏、警觉 | 房间加载、环境音切换 |
| **扫视** | 3-10s | 环顾、识别线索、读文字 | 好奇 | 环境叙事、声学线索 |
| **触发** | 10-20s | 试探环境机关、消耗一发子弹 | 试错 | 音叉/克拉德尼花/涟漪 |
| **战斗脉冲** | 20-40s | 短促遭遇战、读前摇、管理热量 | 紧张 | StarChart + HeatSystem + Enemy AI |
| **清场与余波** | 40-55s | 拾取、读取掉落图谱、停顿呼吸 | 遗憾、满足 | 伤害管线、掉落系统 |
| **过门** | 55-60s | 站在门前、回望、推门 | 决意 | RoomManager + Door |

**设计含义**：
- **战斗不是主体，是"压力脉冲"**：一个 60 秒 Loop 里战斗只占 30%，其余 70% 是探索和情绪过渡
- **每个 Loop 必须有一次"停顿"**：节拍 40-55s 的呼吸感是防止玩家疲劳的关键
- **房间门是 Loop 的自然分界**：这也是为什么 RoomManager 的 authoring 权重这么高

### 1.3 一句话 Loop 描述 ✅ **已锁定**

> **"在大静默的星球上考古，用敌人的思维方式开火。"**（21 字）

这是 Project Ark 的官方一句话定位，同时是团队内部的判断基准。

**如何使用这句话做决策**：

任何功能提案必须能答出"这个功能强化了这句话的哪个动词？"——

| 关键词 | 对应体验 | 强化该词的功能示例 |
|--------|----------|---------------------|
| **大静默** | 世界是空的、低频的、有敬畏感 | 环境音设计、留白关卡、禁用小地图 |
| **星球** | 有地域感，不是抽象关卡 | Tilemap 异星地貌、声学气候、层级纵深 |
| **考古** | 慢、读、拼合、理解 | 环境叙事碎片、声学线索、克拉德尼文阅读 |
| **敌人的思维方式** | 敌人不是靶子，是可被理解的"他者" | 敌人语言系统、AI 行为 Telegraph、异质感知 |
| **开火** | 仍然是动作游戏，不是点击冒险 | 星图战斗、热量管理、读前摇窗口 |

**反向用法（更重要）**：如果一个功能提案无法映射到这 5 个关键词之一，它就该被否决——即使它看起来"有趣"。

**历史候选（已否决，见附录 C）**：
- "驾驶飞船在死寂的星球上，读懂异类的语言并把它编织成武器。"（26 字，过长、偏设定）
- "一个你必须先听懂、才能开火的银河城。"（17 字，缺少"考古"的考据动作感）

## 2. 三根支柱（Three Pillars）

> Core Loop 在**单次 60 秒**尺度上定义"玩家在做什么"，但玩家要持续玩 20~40 小时。
> 三根支柱回答的是"玩家为什么持续玩下去"——它们是**长期动机**，不是单局体验。

### 2.1 World Bible 的三根设计支柱 ✅ **已锁定**

直接继承自 `World_Bible.md` 第 0.2 节，不再讨论：

| # | 支柱 | 定义 | 游戏内具象化 |
|---|------|------|--------------|
| I | **理解即武器** | 玩家的核心进步不是"变强"，而是"变得更理解" | 星图图谱解锁、敌人行为模式识别、异类语言阅读 |
| II | **异质感知** | 六大种族不是"长得奇怪的人类"，每种文明有完全不同的时空观 | 鸣钟者的波形 UI、因果倒置者的倒叙对话、活体图书馆的气味色块 |
| III | **有代价的存在** | 一切行动都有不可逆或有意义的代价 | 热量代价、船员永久死亡、资源路线分岔、Boss 叙事闭环 |

### 2.2 Core Loop 的三根**玩法支柱** 🔵 **推荐版（从 Design Pillars 派生）**

> Design Pillars 是**设计哲学**，玩法支柱是**可被策划/程序落地的具体机制**。

| # | 玩法支柱 | 对应 Design Pillar | 60 秒循环中的体现 | 长期循环中的体现 |
|---|----------|---------------------|------------------|------------------|
| A | **读懂式战斗**<br>(Read-to-Win Combat) | 支柱 I：理解即武器 | 敌人攻击前摇必有 Telegraph，玩家必须读懂才能开窗 | 每种族敌人有独特的"语言"，玩得越久越能一眼看穿 |
| B | **环境即叙事**<br>(Environment as Text) | 支柱 II：异质感知 | 房间内的视觉/声学元素本身是可阅读的信息 | 玩家对世界的认知随探索逐步拼合，不依赖过场动画 |
| C | **选择即消耗**<br>(Choice as Consumption) | 支柱 III：有代价的存在 | 每次开火消耗热量、每次转向消耗窗口 | 每件部件的装备意味着另一种部件的放弃；路线选择不可逆 |

**这三根玩法支柱的重要性**：
- 任何新功能设计，必须能答出"这个功能强化了 A/B/C 中的哪一根？"
- 如果一个功能三根都没强化，它就不该做——即使它看起来"有趣"
- 这也是本文档最实用的"尺子"

### 2.3 支柱互相的张力关系 🔵 **推荐版**

三根支柱不是并列的，它们互相拉扯，正是这种张力创造了 Project Ark 的独特手感：

```
         A (读懂式战斗)
        ╱  ╲
       ╱    ╲
      ╱      ╲
     ╱        ╲
  B ──────────── C
  (环境即叙事)  (选择即消耗)

  A-B 张力：玩家想停下来读环境，但战斗压力逼他快速决策
  A-C 张力：玩家想多开火，但热量和机会成本约束他
  B-C 张力：玩家想探索所有分支，但路线选择的不可逆让他必须取舍
```

**设计含义**：
- 如果某个设计只满足一根支柱而削弱了另外两根，它一定是坏设计
- 例：给玩家装一个"无限热量"模式——满足 A、破坏 C，**否决**
- 例：给玩家加一个"任务标记"系统——方便 A、破坏 B，**否决**（World Bible 也已禁止）

## 3. MVP 证明（Minimum Viable Proof）

> 做到什么程度，我们就知道 Core Loop 是成立的？
> 这一节是**验收标准**——不是"功能完成清单"，是"玩家在某场 60 秒内是否感受到了我们想让他感受的东西"。

### 3.1 MVP 的判断标准 🔵 **推荐版**

MVP 不是"所有系统都跑通"，而是**找一个新玩家（最好是没读过任何设定的人），让他玩 10 分钟，观察他的三个瞬间**：

| 验收瞬间 | 测试方法 | 成功信号 | 失败信号 |
|----------|----------|---------|----------|
| **第一次"听懂"** | 不告诉玩家音叉树的规则，让他自己发现射击会发出音 | 他开始主动测试不同的树、尝试组合音高 | 他射完就走，没有停顿观察反馈 |
| **第一次"犹豫"** | 让他面对"战斗一个歌唱者"的场景，但歌唱者明显没有威胁 | 他在扣扳机前停顿、调整角度、或选择绕过 | 他一视同仁开火，没有任何犹豫 |
| **第一次"认知升级"** | 他解锁第一个棱镜，把新部件装进星图 | 他主动去挑战刚才打不动的敌人，验证新 build | 他装完就继续推进，没有验证意识 |

**三个瞬间都出现 → MVP 成立，Core Loop 的三根支柱都在工作。**
**任何一个瞬间没出现 → 不是系统没做完，是某根支柱的玩法还没被真正设计进去。**

### 3.2 MVP 的技术验收清单 🔵 **推荐版**

对应到当前项目状态（"场景配置与验证阶段"），MVP 需要具备的最小技术条件：

- [ ] 示巴星 ACT1 前 30 分钟（Z1a → Z1c）可端到端通玩，无硬 bug
- [ ] 一场 60 秒的真实战斗遭遇中，玩家能感受到"读前摇 → 开窗 → 开火 → 热量管理"完整链路
- [ ] 至少一个环境互动（推荐音叉树）在无引导情况下可被玩家自然发现
- [ ] 至少一次星图部件变更，能让玩家在下一场战斗中体感到差异
- [ ] 死亡 → 复活 → 重返战场的流程少于 15 秒，并伴随叙事包装（非纯黑屏）
- [ ] 一次完整关卡退出后，玩家能说出 1 条关于世界观的信息（证明环境叙事起效）

### 3.3 MVP 之后的"可扩展性验证" 🔵 **推荐版**

MVP 通过只证明"一个地方能玩"。要证明**整个项目能做完**，还需要一轮"第二个房间测试"：

> 拿已完成的 Z1a 为模板，**用 1 天时间**产出一个全新的 Z1c 变体房间（不同敌人、不同环境机关、不同线索）。
>
> 如果做不到在 1 天内产出第二个可玩房间，说明 authoring 主链还没成熟，继续开工会指数爆炸。

这条测试回答的是 Implement_rules.md 里 Ship/VFX 章节提过的那个核心问题：

> **可扩展的标准不是"这次终于跑起来了"，而是"下次同类需求还能快、还能稳"。**

## 4. Core Loop 与现有模块的对应关系 ✅ **已锁定（派生自当前代码实现）**

> 这一节把 Core Loop 的抽象层与 `Ark_MasterDesign.md` 第 3 节列出的已实现模块对齐，方便工程师判断"我负责的模块服务 Loop 的哪一段"。

| 60 秒节拍 | 负责模块 | 关键脚本 |
|-----------|----------|----------|
| 进入房间 | 关卡 | `Room`, `RoomManager`, `Door`, `Cinemachine Confiner` |

| 清场拾取 | 战斗 + Core | 投射物家族、`DamagePayload`, `DamageCalculator`, 掉落（待补） |
| 过门 | 关卡 | `Door`, `CheckpointSystem`, `WorldProgressManager` |

**空白项说明**：
- "声学线索系统"、"环境互动（音叉/涟漪）"、"掉落系统" 当前未完成——这些是 MVP 前最该补的短板

## 附录 C：v1.0 定稿时被否决的候选（历史存档）

> 保留这些被否决的候选是为了：未来如果有人提出"我们当初为什么不选 A/C？"时，有据可查；同时为 v2.0（如果将来需要）提供重新审视的起点。

### C.1 被否决的 60 秒主轴候选

#### 候选 A：战斗主轴 — "读懂，再开火"（Read, then Fire）

- **核心定义**：60 秒 = 一场 5~8 秒的战斗遭遇 × 若干次 + 间隙的热量管理与走位
- **游戏定位**："一个讲究读懂敌人语言的 Twin-stick 肉鸽/魂味射击游戏"
- **支持理由**：已完成的模块里战斗系统最重；示巴星教学前 32 分钟围绕射击展开；World Bible 对战斗情感描写最具体
- **否决理由**：
  1. 与 World Bible 的"庄严的孤独"基调冲突
  2. 会让项目被归类为"又一个 Twin-stick Roguelite"，失去差异化
  3. Twin-stick 射击的天花板被 Hades / Enter the Gungeon 卡住，技术优势无法转化为市场优势

#### 候选 C：构筑主轴 — "采集，再编织"（Harvest, then Weave）

- **核心定义**：60 秒 = 一次小型采集循环（战斗/探索获取图谱 → 掩体处理决策 → 带着微调 build 继续推进）
- **游戏定位**："一个包裹着 Twin-stick 外壳的 build 驱动 RPG"
- **支持理由**：星图系统是最独特的设计资产；能区隔 Hades 式的随机掉落构筑；天然解决"为什么要打这场战斗"的动机问题
- **否决理由**：
  1. 60 秒内完成"采集→编织→验证"对 UX 要求过高，当前星图是全屏大界面
  2. 容易陷入"为了构筑而构筑"，让战斗/探索本身失去意义
  3. 示巴星教学前 90 分钟星图出现频率低，与这条主轴节奏不匹配
