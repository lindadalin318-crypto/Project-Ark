## 1. Minishoot' Adventures

### 1.1 一句话结论

Minishoot 的关卡架构本质上是一个非常清楚的五层模型：

`全局常驻层 -> Location 场景层 -> Transition 缝合层 -> Encounter/Lock 节奏层 -> Camera/Biome 引导层`

它的强项不在“某个房间设计得多花”，而在于它把开放世界探索、房间遭遇、门锁回路、镜头调度、性能激活，全部拆成了**可重复 authoring 的标准部件**。这也是它能高效铺出“Top-down 射击银河城世界”的关键。

---

### 1.2 可验证的项目层级

基于解包项目与脚本，Minishoot 的场景层不是一个巨型全地图，而是：

```text
-Main
-GlobalObjects
Cave
CaveArena
CaveExtra
Overworld
Snow
Dungeon1~5
Temple1~3
Tower
```

核心含义：

- `-Main` 负责启动与全局管理。
- `-GlobalObjects` 持有玩家、相机、UI、音频等常驻对象。
- 其余场景都可视作一个个 `Location`。
- 玩家并不是“切换游戏模式”，而是在一个统一系统里，从 Overworld 进入 Dungeon / Temple / Arena。

这意味着 Minishoot 采用的是：

**全局对象常驻 + Location 作为世界分块 + Additive 加载与激活切换**

这比“每个关卡一个完全独立关卡模式”更适合银河城，因为：

- 玩家状态连续。
- 世界状态连续。
- 关卡逻辑复用。
- 门、Boss、遭遇、收集品都能共享统一的状态系统。

---

### 1.3 关卡的基本 authoring 单元：`Location`

`Location.cs` 是 Minishoot 关卡架构最关键的类之一。它不是简单的“场景标记”，而是一个**标准化场景容器**。

从脚本可以看出，一个 `Location` 默认会组织这些内容：

- Tilemap 层：`Fence`、`Hole`、`Wall`、`Water` 等
- 功能分组：`Elements`
- 遭遇分组：`Encounters`
- 装饰分组：`Decoration`
- `Checkpoint[]`
- `CrystalKey[]`
- `ActivationManager`

这非常重要，因为它说明 Minishoot 的关卡不是靠“设计师随便摆东西”，而是靠一个**固定的场景语法**：

- `Elements` 放功能物件
- `Encounters` 放战斗/波次/封门
- `Decoration` 放纯氛围或低成本互动

这个层次带来的好处：

1. 设计师知道东西该摆在哪。
2. 程序知道该从哪一层批量检索。
3. 激活/反激活可以按类别统一处理。
4. 场景越做越大时，不容易烂成无法维护的层级地狱。

这是一个非常值得 Ark 借鉴的点：

**好的关卡架构，首先是一套“场景目录规范”。**

---

### 1.4 世界是如何被缝起来的：`Transition`

`Transition.cs` 体现了 Minishoot 世界图的真正骨架。

每个 `Transition` 至少包含：

- `destinationLocation`
- `interiorDoorIndex`
- `sameLocTransition`
- `TransitionType`
- `direction`
- `biomeTrigger`
- `overrideDestinationId`

它在运行时通过一套显式 ID 建图：

```text
CurrentLocation > DestinationLocation_Index
```

以及反向目标：

```text
DestinationLocation > CurrentLocation_Index
```

这代表 Minishoot 的门/出口不是“传送到某个坐标”，而是一个**有命名规则的世界图节点**。

这套做法的优点非常大：

- 关卡连接关系是显式的，不靠隐式约定。
- 可以支持跨场景连接，也支持同场景内跳转。
- 入口方向、落点偏移、进入类型（如洞口掉落）都可封装在 Transition 中。
- 切换后还能顺手触发 Biome 切换和过场表现。

更关键的是，它把“地理连接”变成了一个可以维护的数据网络，而不是一次性的脚本硬编码。

对 Ark 的意义：

- 你现在的 `Door / Passage / RoomManager` 已经有类似潜力。
- 但 Minishoot 给出的启发是：**门不仅是碰撞器，更是世界图边。**
- 如果未来示巴星要做复杂锁钥、跨区域回路、隐藏捷径，显式的门图结构会非常有价值。

---

### 1.5 Minishoot 的核心关卡语法：开放探索 + 房间关闭遭遇

Minishoot 把战斗遭遇拆成了两种角色：

- `EncounterOpen`
- `EncounterClose`

它们不是简单的“刷怪点”，而是两种不同节奏意图的 authoring 组件。

#### `EncounterOpen` 的作用

`EncounterOpen` 更像“开放区域中的激活器”：

- 玩家进触发区，敌人开始激活
- 或敌人进入可视范围时激活
- 玩家离开、敌人被清空、脱离关注后结束

这类遭遇的本质是：

**不打断探索流，只给局部压力。**

适合：

- 路边敌群
- 开放区域巡逻
- 过道型骚扰战
- 世界仍在“呼吸”的感觉

#### `EncounterClose` 的作用

`EncounterClose` 则是严格意义上的“房间战”：

- 玩家进触发器
- 封门/起栅栏
- 切镜头/改 zoom
- 播战斗音乐
- 按波次刷怪
- 可插 Boss 出场演出
- 清空后恢复并写入 `WorldState`

这套结构说明 Minishoot 并没有把“所有战斗都做成一样的战斗”，而是明确分成：

- **开放战斗**
- **关闭式清算战斗**

这是一个非常成熟的节奏控制意识。

对 Ark 的启发很直接：

- 你的 `EncounterSystem` 不应该只有一种模式。
- 至少要区分：
  - 路径骚扰型开放遭遇
  - 锁门清算型房间遭遇
  - 叙事型半脚本战斗
- 否则示巴星会很容易变成“全都是一样的刷怪房”。

---

### 1.6 门锁与世界状态：Minishoot 为什么能形成回路感

Minishoot 的门和锁不是独立小玩具，而是 `WorldState` 的消费者。

代表组件包括：

- `DoorLocked`
- `BossDoor`
- `Transition`
- `Lock`
- `Unlocker`

它们的共同特点：

- 当前是否开启，最终都回落到 `WorldState`
- 场景重进后可恢复
- 某些锁可以与其他锁同步状态
- 某些门是“第一次打开时有演出”，之后只是处于已开状态

这使得世界形成了典型银河城体验：

- 这里以前过不去
- 这次回来能开了
- 门开过之后成为新捷径
- 击败一个关键战斗后，世界结构发生稳定变化

也就是说，Minishoot 的“回路感”不是来自地图大，而是来自：

**稳定持久的空间改写**

这点和 Ark 非常相关。因为你现在已经有 `LockKeySystem`、`RoomManager`、`WorldProgressManager` 类似方向，下一步关键不是继续堆系统，而是把这些系统放进真实关卡中形成：

- 一次阻挡
- 一次学习
- 一次回返
- 一次捷径兑现

---

### 1.7 Biome 与 Camera 不是装饰，而是局部导演系统

Minishoot 有两个非常轻量但非常高效的“导演级”组件：

- `BiomeTrigger`
- `CameraTrigger`

#### `BiomeTrigger`

功能很简单：

- 玩家进入触发器
- 如果当前 Biome 不同，就切换 Biome
- 可顺手控制音乐触发

这带来的不是“换皮肤”而已，而是：

- 区域身份明确
- 音乐和环境氛围即时分段
- 大地图内部也有“情绪分区”

#### `CameraTrigger`

它做的事情更值得注意：

- 局部调整相机缩放
- 相机临时 stick 到某个 Transform
- 可短暂冻结玩家
- 可清场子弹
- 可播放入场音效

这意味着 Minishoot 把相机从“纯跟随器”升级成了**关卡导演工具**。

对玩家来说，这带来的不是技术感，而是：

- 进某个窄房间时更紧
- 进某个景观点时更稳
- 进某个战斗段时更聚焦
- 进某个关键触发点时更有戏剧性

这对 Ark 尤其重要，因为你现在正在做示巴星可玩切片。  
示巴星如果想真正成立，必须很早就把“空间情绪控制”做出来，而不是只靠美术图块。

### 1.8 性能与大地图成立的真正原因：`ActivationManager`

`ActivationManager.cs` 是 Minishoot 很容易被忽略、但其实极其关键的长处之一。

它会按 `Location` 批量收集各种可激活对象：

- 可破坏物
- 锁与门
- Transition
- CameraTrigger
- Deactivable
- Torch
- HiddenArea
- EncounterOpen / EncounterClose / EncounterHidden
- Light

然后按距离与可见性做激活/反激活：

- 靠近时启用
- 远离时停用
- 遭遇进行中可阻止反激活
- 已销毁对象不会被错误重新启用

这套设计的价值不是“优化帧数”这么简单，而是：

**它允许设计师放心地把世界做大。**

因为没有这个系统，大地图关卡一旦密度上来：

- 设计师会害怕多摆东西
- 每加一个场景装饰都像在透支性能
- 遭遇战、可破坏物、灯光、隐藏区会彼此拖累

而有了 `ActivationManager`，关卡 authoring 才有空间呼吸。

对 Ark 的启发：

- 你已经在做房间系统和单场景 Tilemap 架构。
- 越往后，越需要一个“按 Room/Chunk 批量启停”的标准机制。
- Minishoot 给你的提醒是：**关卡架构必须包含运行时激活策略，否则再好的布局都难规模化。**

### 1.9 Minishoot 的实际构建流程推断

结合场景命名、`Location` 结构和脚本职责，Minishoot 的设计师/关卡搭建流程大概率是这样的：

#### Step 1：建立一个 Location 场景

- 比如 `Overworld`、`Cave`、`Dungeon2`
- 这个场景本身就是一整个关卡分区

#### Step 2：按固定根节点分层

- Tilemap 层
- `Elements`
- `Encounters`
- `Decoration`
- `Checkpoints`
- `Transitions`
- `BiomeTriggers`
- `CameraTriggers`

#### Step 3：先搭世界骨架，再填战斗

- 先确定玩家怎么走
- 哪些出口连到哪里
- 哪些门是锁、哪些门是捷径
- 哪些区域是安全段、哪些区域是清算段

#### Step 4：遭遇 authoring 是局部模块化的

- 要开放骚扰战，就摆 `EncounterOpen`
- 要封门房战，就摆 `EncounterClose`
- 要 Boss，就把 Boss 挂到对应 Encounter 下

#### Step 5：用 Trigger 调情绪，不用重写逻辑

- Biome 变换靠 `BiomeTrigger`
- 镜头节奏靠 `CameraTrigger`
- 过渡靠 `Transition`

#### Step 6：一切写入世界状态

- 门是否开过
- 遭遇是否完成
- Boss 是否见过/击败
- Checkpoint 到哪

这是一种非常成熟的“拼装式关卡生产方式”。  
它的核心不是工具有多复杂，而是**每个设计目标都对应一个标准部件**。

### 1.10 Minishoot 最值得偷的长处

下面这些，我认为是 Minishoot 最值得 Project Ark 借鉴的部分。

#### 长处 A：把“关卡”做成一套清晰语法，而不是一堆房间

Minishoot 让每个 Location 都遵循相同的结构约定。  
这会大幅降低后期维护、接手、扩张的成本。

#### 长处 B：开放探索与封闭遭遇分离

它知道什么时候该让玩家持续流动，什么时候该把玩家关起来做清算。  
很多银河城失败，就是因为这两种节奏没有被明确区分。

#### 长处 C：门与状态形成世界回路

真正的银河城不是地图大，而是“我回来时世界变了”。  
Minishoot 很会用持久门锁和捷径兑现这种感觉。

#### 长处 D：轻量 Trigger 驱动重体验

Biome、Camera、Transition 都是轻量组件，但可以显著改变玩家感受。  
这是一种很高 ROI 的关卡导演思路。

#### 长处 E：性能策略内建到关卡架构中

`ActivationManager` 说明它不是做完关卡再补优化，而是一开始就把“世界怎么被激活”纳入架构。

#### 长处 F：统一词汇贯穿 Overworld / Dungeon / Arena

它不是为不同区域发明不同的系统，而是用同一套 Location/Transition/Encounter 语法覆盖所有空间类型。  
这让项目既统一，又有扩展性。

---

### 1.11 哪些东西不该直接照抄

Minishoot 很强，但不是所有做法都适合 Ark 直接照搬。

#### 不该直接照抄 A：静态单例 + 全局状态的高度耦合

Minishoot 大量依赖静态访问与全局 `WorldState`。  
这对小中型项目很快，但对 Ark 这种系统野心更大的项目，后期会变脆。

#### 不该直接照抄 B：场景级 Location 直接映射到 Ark 的每个房间

Ark 现在更接近“单场景 + Tilemap 房间系统”。  
所以 Minishoot 的 `Location` 更适合作为你的 `Room Cluster` 或 `Functional Chunk` 的思想参考，而不是 1:1 变成 Unity Scene。

#### 不该直接照抄 C：用大量布尔世界状态无限扩张

`WorldState.Get("xxx")` 很灵活，但规模大了会进入维护地狱。  
Ark 更适合保留显式数据结构和类型化状态。

### 1.12 对 Project Ark 的直接映射建议

如果把 Minishoot 的长处翻译到 Ark，我建议这样映射：

| Minishoot | Project Ark 对应方向 | 借鉴方式 |
|---|---|---|
| `Location` | `Room` / `Room Cluster` / 区域功能块 | 统一房间根结构与子节点规范 |
| `Transition` | `Door` / `Passage` / 房间边连接 | 显式世界图边，支持双向与捷径 |
| `EncounterOpen` | 开放骚扰遭遇 | 不打断探索，只增加局部压力 |
| `EncounterClose` | 锁门清算遭遇 | 用于关键房、Boss 前厅、战斗教学 |
| `DoorLocked` / `BossDoor` | 锁钥门 / Arena 封门 / 清算门 | 全部接入持久世界进度 |
| `BiomeTrigger` | Room mood / ambience / sound profile 切换 | 做局部情绪分区 |
| `CameraTrigger` | 房间镜头约束 / 戏剧化镜头 | 用最小成本强化空间体验 |
| `ActivationManager` | Room/Chunk 激活系统 | 为单场景大地图做性能护栏 |

### 1.13 对示巴星切片的具体启发

如果只看你当前最需要做的“可玩关卡切片”，Minishoot 给你的最有用启示不是“地图风格”，而是下面这五条：

#### 1. 先定骨架，再填内容

不要先做很多敌人和机关。  
先把：

- 入口
- 关键房
- 门
- 回路
- 安全点
- 第一次能力获得点

这些骨架确定。

#### 2. 遭遇必须分层

示巴星前 10-15 分钟至少要区分：

- 路上骚扰战
- 明确的锁门教学战

两者不能做成一种味道。

#### 3. 镜头和环境切换要及早介入

示巴星不是普通射击图。  
如果没有镜头收束、环境音切换、局部空间情绪变化，玩家很难真正感到“这颗星球在用声音塑形”。

#### 4. 门的解锁要产生真正的回返收益

哪怕是首个切片，也最好有一条“小捷径被打开”的体验。  
这样玩家才能感觉这不是线性关卡，而是银河城。

#### 5. 性能与 authoring 规范要尽早定

示巴星将来一定会有很多：

- 声学机关
- 黑水 hazard
- 叙事触发点
- 小型生态演出

越晚定节点规范和房间激活策略，越容易返工。

### 1.14 对 Minishoot 的最终评价

如果只从“关卡是如何被构建出来的”角度看，Minishoot 最值得学习的不是它某个神来之笔，而是它的**工程化关卡思维**：

- 世界分块清楚
- 连接规则清楚
- 房间节奏清楚
- 导演工具轻量
- 状态持久稳定
- 性能策略内建

它给 Ark 的最大启示可以浓缩成一句话：

**先定义关卡语法，再大规模生产内容。**

---

### 1.15 更深一层：Minishoot 的“世界推进”是怎么和关卡绑定的

前面的分析已经说明它有 `Location / Transition / Encounter / Camera / Biome` 这套语法。  
继续往里看，会发现 Minishoot 真正厉害的地方在于：

**它把“世界推进”设计成关卡构建的一部分，而不是后挂的剧情脚本。**

这个绑定主要靠三样东西：

- `WorldState`
- `EncounterWave`
- `TransitionDungeonOutside`

#### `WorldState`：一切空间变化的持久记忆

`WorldState` 是一个全局布尔字典，看起来朴素，但它承担了整个世界的结构记忆。

从代码可以确认，它至少在管理这些层面：

- 某个遭遇是否已完成
- Arena 某一关是否通过/是否无伤
- 是否到达某个关键区域（如 `OverworldReached`）
- 某个 Boss 是否击败
- 某个 NPC 是否解救
- 某些关键对话是否已看过

这意味着 Minishoot 的空间不是“每次进来都原样重播”，而是一个持续被玩家行为改写的世界。

这会直接带来三种经典银河城体验：

1. **区域身份变化**
   某个地方在没打通前是阻挡，打通后变成通路。

2. **内容密度变化**
   有些触发器、Boss 门、Dungeon 入口会随着状态变化而改变表现。

3. **回返意义**
   回头不是重复走旧路，而是在看“旧路现在变成什么样”。

#### `EncounterWave`：战斗不是“刷一波敌人”，而是有演出顺序的

`EncounterWave` 说明 Minishoot 的房间战也不是一次性把敌人堆满：

- 每个 Wave 有自己的敌人集合
- 支持 Boss 波
- 支持入场显形动画
- 支持碰撞延后开启
- 支持房间清空后再开下一波

这实际上让一个房间里的战斗拥有了：

- 起
- 承
- 转
- 合

也就是说，Minishoot 不是“房间 + 战斗”，而是“**房间内有一个短篇战斗剧本**”。

这对 Ark 很重要，因为 Ark 现在也有 EncounterSystem，但示巴星之后如果想真正做到“战斗房有记忆点”，就不能只停在：

- 刷一批怪
- 清空开门

而应该至少进一步做到：

- 第一次进入发生什么
- 中段是否改空间条件
- 最后一波是否改变战斗语法
- 战后世界是否发生持久改写

#### `TransitionDungeonOutside`：入口本身就是推进可视化

这个类非常值得注意。  
它会根据某个 Dungeon 的 Boss 是否被击败，改变外部入口的视觉与音效表现：

- 环境粒子是否播放
- 门口光效是否存在
- 环境循环音是否启用
- “净化后”的外观是否显示

这说明 Minishoot 把 Dungeon 完成度直接投射回世界表面。

这很聪明，因为它让玩家不需要开菜单，也能“看见自己改变了世界”。

对 Ark 的意义非常大：

- 示巴星如果只有日志说“你改变了这里”，效果会很弱。
- 但如果打完某个节点后，区域声场、黑水高度、建筑共鸣状态、缕拉留下的生命痕迹都发生可见变化，玩家会强烈感到自己真的介入了世界。

---

### 1.16 Overworld 不是“Hub”，而是有层次的导航平面

从 `Overworld.unity` 里的 `destinationLocation` 搜索可以看出，Overworld 并不是单纯的大地图外壳，它实际上承担了世界导航平面的职责。

它连向的目标包含：

- `Cave`
- `CaveExtra`
- `Snow`
- `Temple1~3`
- `Dungeon1~5`
- `Tower`
- `CaveArena`

这说明 Minishoot 的 Overworld 更接近：

**一个聚合多个子空间入口、并通过回返逐步重新解释自身的导航平面。**

其价值不只是“提供旅行”，而是：

- 让玩家不断从一个稳定的世界底图上重新理解整体结构
- 把复杂空间拆成“外部导航 + 内部专属区域”两层
- 给回返、捷径、入口状态变化提供承载面

这也是为什么 Minishoot 的世界虽然分场景，但仍有强烈的一体感。

对 Ark 来说，对应的启示是：

- 你未来不一定要做一个“传统大地图 Overworld”
- 但一定需要一个**可反复重读的导航层**
- 示巴星内部可以通过“大区骨架 + 深层专属区”的方式，制造类似效果

也就是说，玩家需要不断回到一个自己熟悉、但不断变化的空间底图上，才能真正建立“银河城脑内地图”。

### 1.17 Biome 在 Minishoot 中不是美术标签，而是“区域状态机”

前面我们提到 `BiomeTrigger` 很轻量。  
继续看 `BiomeManager` 会发现，它触发的远不止音乐：

- `Music.PlayBiome(biome)`
- `AmbientSfx.Play(biome)`
- `LightManager.Instance.UpdateIntensity(biome, duration)`
- `ParticlesAmbientManager.Instance.Set(biome)`
- Bloom / Vignette / WhiteBalance / ColorAdjustments 等后处理联动

换句话说，Biome 在 Minishoot 里实际承担的是：

**局部环境导演预设**

一旦玩家踏进某个触发器，整个区域的：

- 听觉
- 光照
- 粒子
- 后处理色调

都会被切换到一个新的“区域身份”。

这会带来一个很成熟的关卡效果：

- 场景未必需要大量重做
- 但区域气质可以明显分段
- 一个大场景内部仍能形成多个情绪章节

对 Ark 的直接借鉴是：

示巴星后续完全可以做自己的“Biome-like 区域状态预设”，例如：

- `静默沼泽态`
- `高共鸣峡谷态`
- `教堂渐强态`
- `孵化深渊警报态`

每一种不只是改 BGM，而是联动：

- 环境音层
- 光强
- VFX 密度
- HUD 噪声程度
- 热量散逸修正

这样你的世界观机制才会真正“落到空间上”。

---

### 1.18 Checkpoint 的设计说明了一件事：存档点也是空间语法的一部分

`Checkpoint.cs` 看起来很简单，但它体现了 Minishoot 的一个成熟观念：

**Checkpoint 不是菜单系统，它是关卡结构节点。**

它做的事情包括：

- 设置 `Current`
- 写入 `PlayerState`
- 驱动某些全局关键进度（如到达 Overworld）
- 根据世界状态决定自己是否还应存在

这说明 Minishoot 的 Checkpoint 不是纯“复活功能”，而是：

- 进入某个章节的确认器
- 空间推进的里程碑
- 关卡节奏的断句点

这和很多银河城的优秀经验一致：  
好的存档点放置，不只是为了方便，而是为了告诉玩家：

> 这里是一个结构性阶段完成的位置。

对 Ark 来说，这一点非常关键。  
你已经在做 `CheckpointSystem`，但真正需要追问的是：

- 这个点是否是空间节奏的句号？
- 是否是回返路线的基座？
- 是否代表玩家刚学会/刚证明了一件事？

如果只是机械等距离摆点，就很难形成“这是一段完整旅程”的感受。

### 1.19 一个最值得偷的制作方法：用少量标准件重复组合，做出复杂世界

如果把 Minishoot 的关卡生产方式再提炼一层，它最值得学的不是某个代码类，而是这个制作原则：

> **不要为每一段空间重新发明玩法语言。**

它实际是在反复复用少量标准件：

- `Location`
- `Transition`
- `Checkpoint`
- `EncounterOpen`
- `EncounterClose`
- `DoorLocked` / `BossDoor`
- `BiomeTrigger`
- `CameraTrigger`

每个标准件承担清晰职责。  
复杂度不是来自“有 100 套系统”，而是来自：

**标准件在不同空间关系中的组合方式不同。**

这套思想非常适合 Ark 当前阶段，因为你最缺的已经不是系统数量，而是：

- 哪些东西要成为标准件
- 哪些体验要通过组合实现
- 哪些内容不该再定制化过早

我会把它翻译成 Ark 的一句话建议：

**先定义 8-12 个示巴星关卡标准件，再开始大规模造房。**

比如未来可以是：

- 房间入口
- 房间出口
- 锁门清算
- 开放骚扰
- 声场切换器
- 黑水 hazard
- 缕拉痕迹点
- 叙事遗迹点
- 热量压力点
- 镜头/共鸣演出点

只要这些标准件稳定，内容生产速度会快很多，而且整体质感更统一。

### 1.20 直接翻译成 Ark：首个可玩切片的关卡模板

如果把 Minishoot 的强项直接翻译成一个 `Project Ark` 可执行模板，我建议你的首个可玩切片至少按下面的结构来搭。

#### 关卡骨架模板

```text
入口安全段
  -> 开放骚扰段
  -> 第一清算房
  -> 短回报段（Lore / 视觉奇观 / 缓冲）
  -> 能力或认知获得点
  -> 回返捷径开启
  -> 收束安全点 / Checkpoint
```

这个结构的来源本质上就是 Minishoot 的：

- `Transition` 给骨架
- `EncounterOpen` 给压力
- `EncounterClose` 给清算
- `Checkpoint` 给断句
- `Door/Lock` 给回路
- `Biome/Camera` 给情绪变化

#### 对应到示巴星前 10-15 分钟

一个可执行映射大概可以是：

1. **坠落点 / 安全段**
   玩家醒来，学会移动和基础观察。

2. **静默压力段**
   低声环境让热量难以处理，玩家第一次感到“这里的安静不安全”。

3. **开放骚扰段**
   放少量骚扰敌人，不锁门，让玩家知道可以边走边打。

4. **第一清算房**
   用锁门或空间收束让玩家真正理解“战斗=散热窗口”。

5. **环境回报段**
   冰雕 / 花路 / 音叉树 / 回声物件，让世界观不是全靠文本。

6. **缕拉第一次强存在感**
   不一定深对话，但至少让玩家知道“这个世界里有一个主动影响我的存在”。

7. **能力或认知获得**
   比如 `涟漪`、或第一次“声音能开路”的明确确认。

8. **捷径开启**
   返回入口附近或连接到安全点，让玩家体会银河城回路，而不是纯线性推进。

9. **Checkpoint / Safe Room**
   给玩家一个很清楚的“这段玩完了”的句号。

#### Minishoot 给 Ark 的最实用落地原则

如果只留下 5 句对当前开发最有帮助的话，我会留这 5 句：

1. **先定义关卡标准件，再做房间。**
2. **开放骚扰和锁门清算一定要分开。**
3. **门锁和捷径必须写入持久世界状态。**
4. **镜头、环境音、后处理要作为关卡导演工具，而不是上线前点缀。**
5. **每个 Checkpoint 都应该是一个节奏句号，而不是随便插的复活点。**

### 1.21 Minishoot 的房间组织法则：不是“房间并列”，而是“节点分级”

继续结合场景对象命名，可以看到 Minishoot 的空间并不是由等权房间拼成，而是由不同职责的节点构成。

例如：

- `Cave.unity` 中可检出的对象大致有：
  - `EncounterOpen0~23`，约 **24 个开放遭遇**
  - `CaveEncounterClose0~13`，约 **14 个关闭遭遇**
  - 多个命名型 `CameraTrigger`：`CameraTriggerIntro`、`CameraTriggerPrimordial`、`CameraTriggerMuseum`
  - 多个 Biome：`BiomeTown`、`BiomeCave`、`BiomeSanctuary`、`BiomeSewer`
- `Overworld.unity` 中可检出的对象大致有：
  - `EncounterOpen0~73`，约 **70+ 个开放遭遇**
  - `OverworldEncounterClose0~15`，约 **16 个关闭遭遇**
  - 一组明确的大区 Biome：`Forest`、`Desert`、`Swamp`、`Town`、`Junkyard`、`Abyss`、`Cemetery`

从这个比例能看出一个很明确的组织原则：

**开放遭遇是“底噪”，关闭遭遇是“节点”。**

也就是说，Minishoot 的世界组织不是：

- 房间A
- 房间B
- 房间C

而更像：

- 通路节点
- 压力节点
- 清算节点
- 叙事节点
- 枢纽节点

这才是它玩起来既流动又不失控的原因。

#### 可以抽象出的 6 种房间/节点类型

##### 1. 过路节点（Transit Node）

作用：

- 保持移动
- 连接相邻区域
- 不承载太重的信息

典型特征：

- 可能有少量 `EncounterOpen`
- 很少锁门
- 常承担 Transition 或 Biome 切换

##### 2. 压力节点（Pressure Node）

作用：

- 在不打断流程的情况下增加消耗和警惕

典型特征：

- 1-3 组开放敌人
- 可边打边走
- 常和地形、视野、移动约束叠加

##### 3. 清算节点（Resolution Node）

作用：

- 用一场明确的封闭战斗验证玩家是否掌握了当前区段规则

典型特征：

- `EncounterClose`
- 封门/镜头收束/音乐切换
- 波次或 Boss

##### 4. 回报节点（Reward Node）

作用：

- 在紧张后给玩家认知回报、资源回报或氛围缓冲

典型特征：

- Checkpoint
- 宝箱/升级/叙事物件
- 景观房

##### 5. 锚点节点（Anchor Node）

作用：

- 帮助玩家建立脑内地图

典型特征：

- 明显地标
- 特殊 CameraTrigger
- 独特 Biome 切换
- 命名明显的演出主题（如 `Museum`、`Primordial`）

##### 6. 回路节点（Loop Node）

作用：

- 把“单向推进”转化成“结构性掌控”

典型特征：

- 解锁门
- 反向通路
- 回到旧 Checkpoint 的捷径

#### 这套节点分级对 Ark 的意义

你做示巴星切片时，最好不要说“我要做 8 个房间”，而要说：

- 我要 2 个过路节点
- 2 个压力节点
- 1 个清算节点
- 1 个回报节点
- 1 个锚点节点
- 1 个回路节点

这样搭出来的空间天然更像“关卡”，而不是“地图碎块”。

### 1.22 Minishoot 的节奏法则：探索流必须显著大于清算流

从 `Cave` 和 `Overworld` 的 Encounter 命名数量看，Minishoot 很明显遵循一个节奏比例：

> **开放探索流占大头，锁门清算流占小头，但锁门节点决定记忆点。**

这背后至少有 4 条节奏法则。

#### 法则 A：敌人密度高，不等于清算密度高

Overworld 里有大量 `EncounterOpen`，但 `EncounterClose` 相对少得多。  
说明 Minishoot 追求的是：

- 玩家一直感觉世界里有敌意
- 但不是每几十秒都必须停下来“被迫解决”

这是保持开放世界流动感的关键。

如果把每一段小遭遇都做成封门清算，世界会非常疲惫。

#### 法则 B：清算节点必须服务于“阶段结论”

`EncounterClose` 的价值不是难，而是它通常代表：

- 一个路径阶段完成
- 一个房间规则被证明
- 一个局部危险被清空
- 一个门或世界状态将被改写

也就是说，清算战不是日常噪声，而是**结构性标点符号**。

#### 法则 C：镜头与 Biome 变化用于“换章节感”

从 `CameraTriggerIntro`、`CameraTriggerMuseum`、`BiomeTown`、`BiomeSewer` 这些命名就能看出，Minishoot 很清楚：

- 节奏变化不只靠敌人
- 更靠空间身份和演出切换

所以即使玩家还在同一大区域内，也会感受到“我进入了新段落”。

#### 法则 D：回报要紧跟在证明之后

一个成功的清算节点后，通常不应该立刻再给第二个高压清算。  
Minishoot 的结构暗示它更倾向：

```text
压力
 -> 清算
 -> 短回报 / 缓冲
 -> 再进入下一个压力段
```

这能让玩家感到：

- 我刚刚赢得了一点掌控
- 游戏承认了我的进步
- 我有时间重组注意力

这对 Ark 很重要，因为你的系统层压感更强。  
如果示巴星把“热量压力 + 敌人 + 世界观信息 + 平台风险”连续高密堆叠，很容易把玩家压成纯消耗态。

### 1.23 Minishoot 的房间连接法则：每次连接都在回答一个问题

Minishoot 的 `Transition` 很多，不是为了“传送得多”，而是为了让每次连接都回答一个明确问题。

我把它总结成 5 类连接关系：

#### 连接 1：推进连接

问题：

> 我还能往前走吗？

用途：

- 当前主路线推进
- 通常连接压力段或新区域

#### 连接 2：回返连接

问题：

> 我如果回头，会不会更快？

用途：

- 捷径
- 重新连接旧安全点

#### 连接 3：能力连接

问题：

> 这里先记住，以后拿到能力再回来。

用途：

- 银河城能力门
- 暂时性视觉钩子

#### 连接 4：挑战连接

问题：

> 你现在有没有资格进入更高压内容？

用途：

- Arena / Boss 前厅 / 支线挑战房

#### 连接 5：身份连接

问题：

> 你刚刚进入了什么类型的地方？

用途：

- Biome 切换
- CameraTrigger
- 空间章节分割

#### 对 Ark 的直接启发

示巴星前 15 分钟切片里，哪怕只做 4-6 个连接点，也最好让每个连接点承担明确语义，而不是全都只是“下一个门”。

例如：

- 一个是纯推进连接
- 一个是拿到 `涟漪` 之后反开的回返连接
- 一个是显式的能力门预告
- 一个是进入缕拉痕迹区域的身份连接

只要连接有语义，地图就会开始“说话”。

### 1.24 Minishoot 的首段关卡节拍，翻译成可抄的模板

如果我把 Minishoot 的前段组织方法翻译成一个可以直接照着搭的节拍模板，它大概是这样的：

```text
0. 安全着陆 / 恢复控制
1. 轻探索，建立空间阅读
2. 第一次低风险骚扰
3. 小回报，证明前进方向正确
4. 第一次明确门槛或封闭战
5. 战后回报（能力、剧情、景观、NPC）
6. 给出一个新的目标方向
7. 打开一条回返或捷径
8. 在安全点收束
```

这套节拍最厉害的地方在于：

- 每一步都不复杂
- 但组合起来就很像完整冒险

#### 适配示巴星前 15 分钟的版本

```text
0. 坠机苏醒
1. 在死寂冰原学会移动与观察
2. 第一次感受静默导致热量不安全
3. 一小波开放骚扰敌人
4. 看到缕拉痕迹 / 花路 / 冰雕
5. 第一场锁门清算，证明“战斗=散热窗口”
6. 获得涟漪或第一次声学交互确认
7. 打开回返捷径
8. 到达第一个安全点 / Checkpoint
```

注意这里最关键的不是事件本身，而是**节拍顺序**。  
Minishoot 的长处就在于它很少让事件互相打架。

### 1.25 可直接照着搭的“示巴星房间表”版本

下面这版更具体，适合你之后直接翻成关卡表或 room list。

| 房间/节点编号 | 类型 | 目标 | 建议内容 | 应验证的体验 |
|---|---|---|---|---|
| `SH-SLICE-R01` | 入口安全节点 | 建立基调 | 坠机残骸、可读地标、低敌压 | “我掉进了一个死寂但有秩序的世界” |
| `SH-SLICE-R02` | 压力节点 | 引入静默压力 | 低声场、过热上升、少量可绕敌人 | “这里的安静不是安全，是风险” |
| `SH-SLICE-R03` | 锚点节点 | 建立记忆点 | 冰雕 / 花路 / 音叉树 / 远景中的缕拉影子 | “这个世界有历史，也有某个存在在看着我” |
| `SH-SLICE-R04` | 清算节点 | 验证核心 loop | 锁门、1-2波敌人、战斗散热明显 | “主动制造噪音反而让我活下来” |
| `SH-SLICE-R05` | 回报节点 | 给认知奖励 | 小 Lore、短安全窗、工具获取 | “我不只是打赢了，我学会了一件事” |
| `SH-SLICE-R06` | 回路节点 | 银河城兑现 | 开门回返、缩短路线、回到前面区域边缘 | “世界被我永久改了一点” |
| `SH-SLICE-R07` | 收束节点 | 形成段落句号 | Checkpoint / Safe Room / 缕拉首次稳定互动 | “这是完整的一段旅程，不是半成品试玩” |

这张表本质上就是把 Minishoot 的空间方法翻译成 Ark 可执行版本。

---

### 1.26 一条最重要的结论：不要从“内容量”思考切片，要从“节奏闭环”思考切片

Minishoot 最值得你学的具体方法，不是多做房间，也不是多做系统，而是：

> **一个切片是否成立，取决于它有没有完成一条节奏闭环。**

这个闭环至少应包含：

- 进入
- 施压
- 证明
- 回报
- 回路
- 收束

只要这 6 件事完整，哪怕地图不大，玩家也会觉得“这是一段真的游戏内容”。

如果缺其中两三项，就会很容易变成：

- 只是一个测试场
- 只是战斗拼盘
- 只是美术走廊
- 只是系统样品

所以，Minishoot 给 Ark 当前最务实的建议可以浓缩成一句话：

**先做一个完成节奏闭环的示巴星切片，再考虑扩张区域规模。**

---

### 1.27 Minishoot 机制词典：可互动部件、机关、门锁、隐藏区、传送

下面这部分专门把 Minishoot 中与关卡 authoring 直接相关的互动件整理成一套词典。  
目标不是罗列所有脚本，而是回答 5 个问题：

1. 这个部件在玩家体验中扮演什么角色？
2. 它是怎么被触发的？
3. 它是否写入持久状态？
4. 它更偏“空间结构”还是“瞬时演出”？
5. Ark 能借它的哪一层思想？

---

#### A. 门锁与开门部件

##### 1. `Transition`

**本质**：世界图边 + 入口点 + 过渡演出器

**触发方式**：

- 玩家碰撞进入
- 若已解锁或本身无需解锁，则开始转场

**做的事**：

- 记录最后一次 Transition
- 临时停用玩家
- 清掉敌方子弹
- 播进入演出
- 加载目标 Location
- 把玩家放到目标入口点前方固定偏移
- 切换 Biome

**是否持久化**：

- 若本身是 `Lock` 子类且可解锁，则写入 `WorldState`

**关卡意义**：

- 不是“传送点”而已，而是所有空间连接的标准件

**Ark 可借鉴点**：

- 房间边连接要成为标准组件
- 入口方向、重定位、相机、环境切换最好内聚在同一部件语义中

##### 2. `DoorLocked`

**本质**：最基础的“已开/未开”门

**触发方式**：

- 调用 `Unlock()`

**表现方式**：

- 直接调整门上下两部分的 sprite size，形成开门视觉

**是否持久化**：

- 是，写入 `WorldState`

**关卡意义**：

- 提供最朴素的“阻挡 -> 打开 -> 永久捷径”

**Ark 可借鉴点**：

- 简单门不一定要很复杂，重要的是它接入世界状态

##### 3. `BossDoor`

**本质**：带独立演出的 Boss 门

**触发方式**：

- 调用 `Unlock()`

**表现方式**：

- 禁用玩家
- 播门打开特写演出
- 解锁后恢复控制

**是否持久化**：

- 是

**关卡意义**：

- 它告诉玩家：这不是普通门，这是章节边界

**Ark 可借鉴点**：

- Boss 门与普通门应有不同的仪式感级别

##### 4. `StrongDoor`

**本质**：多段部件组成的重型机关门

**触发方式**：

- 调用 `Unlock()`

**表现方式**：

- 多个 `LockPart` 分段旋转/位移
- 粒子、震屏、音效、玩家表情
- 可连带解锁第二目标
- 可连带摧毁某个目标对象

**是否持久化**：

- 是

**关卡意义**：

- 这是“世界结构被改写”的重型版本，不只是门开了，而是空间秩序被重排

**Ark 可借鉴点**：

- 对重大门禁、封印、坍塌、机械门，最好做成“多件联动”的重量级打开，而不是单 sprite 切状态

##### 5. `TriggerOpenStrongDoorPatch`

**本质**：一种很有意思的“门状态重置补丁”

**触发方式**：

- 玩家进入触发器

**做的事**：

- 如果强门已开，则把其 `WorldState` 改回 false
- 强制门恢复为关闭状态

**关卡意义**：

- 说明 Minishoot 某些门不是纯永久捷径，而是可以被局部关卡逻辑回收重用

**Ark 可借鉴点**：

- 不是所有门都该永久打开
- 某些戏剧化场景可以允许“结构临时重封”，前提是语义明确

---

#### B. 解锁器 / 钥匙 / 条件触发部件

##### 6. `Unlocker`

**本质**：靠接触触发、消耗钥匙或条件后解锁目标的标准件

**支持的钥匙类型**：

- `None`
- `Normal`
- `Boss`
- `Scarab`
- `Darker`
- `FinalBoss`

**可做的事**：

- 判断玩家是否持有某类钥匙
- 消耗普通/Boss 钥匙
- 支持 group 联动（全部按下才开）
- 支持一次开两个锁
- 支持镜头飞到目标位置再开门
- 支持隐藏信号提示

**是否持久化**：

- 目标锁会写入 `WorldState`
- 某些 Demo 按钮还会记 Pressed 状态

**关卡意义**：

- 它不是“钥匙孔”这么简单，而是一个条件解锁器模板

**Ark 可借鉴点**：

- 你的锁钥系统未来也应该尽量做成统一的条件解锁器，而不是每种门各写一套脚本

##### 7. `CrystalKey` / `BossKey` / `KeyUnique`

**本质**：三种不同语义的钥匙拾取物

**区别**：

- `CrystalKey`：普通地牢钥匙
- `BossKey`：Boss 门钥匙
- `KeyUnique`：唯一钥匙，直接把某个 `KeyUse` 标记设为 true

**是否持久化**：

- 都会写入 `WorldState`

**关卡意义**：

- Minishoot 把“钥匙”分成消耗型与永久型两类

**Ark 可借鉴点**：

- 未来示巴星乃至全项目的锁钥，最好尽早区分：
  - 消耗资源型
  - 世界权限型
  - 剧情唯一物型

##### 8. `UnlockerTorch`

**本质**：火炬点亮链式解谜 + 解锁器 + 轻演出系统

**触发方式**：

- 被点亮后记入 `WorldState`

**可做的事**：

- 火炬组全部点亮才解锁目标
- 点亮一支后镜头 travel 到下一支火炬
- 全亮后镜头 travel 到目标门
- 可同时解锁两个目标
- 会为玩家恢复少量能量

**是否持久化**：

- 是，火炬点亮状态本身就是持久状态

**关卡意义**：

- 这是非常典型的“环境谜题变成结构解锁”的做法

**Ark 可借鉴点**：

- 示巴星完全可以有“声学节点链”或“共鸣柱链”作为对应物
- 关键不是火炬本身，而是：
  - 局部完成
  - 镜头引导下一个目标
  - 最终改写空间结构

##### 9. `Torch`

**本质**：点亮后的视觉/声音状态承载体

**做的事**：

- 播激活粒子
- 播火焰循环
- 开启循环音

**关卡意义**：

- 把“状态改变”做成可感知的环境资产，而不只是 bool

**Ark 可借鉴点**：

- 任何被激活的声学装置，都应该有稳定的环境反馈层

---

#### C. 隐藏区与伏击类部件

##### 10. `HiddenArea`

**本质**：可穿透的“视觉遮挡层”

**触发方式**：

- 玩家进出 trigger

**做的事**：

- 进入时把 Sprite 的 `_Fade` 动态淡出
- 离开时恢复

**关卡意义**：

- 它不是秘密本身，而是“秘密入口的可读性工具”

**Ark 可借鉴点**：

- 示巴星未来若有前景遮挡、菌幕、声雾、黑水帘幕，这种“进入即淡出”的轻机关很好用

##### 11. `EncounterHidden`

**本质**：隐藏伏击点 / 埋伏遭遇 / 条件出现敌群

**触发方式**：

- 玩家进入隐藏碰撞区
- 可叠加 `MapObjective` 完成条件
- 可有 `spawnChance`

**做的事**：

- 激活一组 `EncounterWave`
- 若玩家脱离视野和区域，可重置
- 清空后可记入 `WorldState`
- 也可以设计成“先锁住，后 `Unlock()` 才可触发”

**关卡意义**：

- 这是开放探索里很有价值的“次级惊喜/惩罚节点”

**Ark 可借鉴点**：

- 你未来完全可以把某些黑水裂隙、共鸣残响、缕拉痕迹做成“可触发的隐藏遭遇”
- 这样世界会显得更像活体，而不是全都摆在台面上

---

#### D. 可破坏环境部件

##### 12. `Destroyable`

**本质**：通用可伤害/可销毁核心

**支持的能力非常多**：

- 指定可被哪些 tag 伤害
- 独立 HP
- 恢复/无敌/永久销毁
- 受击恢复窗口
- 暴击/特殊伤害系数
- 死亡时粒子与音效
- 死亡后关闭目标对象
- 死亡后联动火炬
- 写入 `WorldState`

**关卡意义**：

- 这是 Minishoot 把“敌人、机关、环境可破坏物”部分统一在一套受击语法里的核心基础

**Ark 可借鉴点**：

- 你的 `DamagePayload` / `IDamageable` 路线是对的
- 长期应争取让“可破坏环境”也尽量吃同一套伤害语法

##### 13. `WallDestroyable` / `RockDestroyable`

**本质**：持久破坏型墙体/岩石障碍

**表现方式**：

- 多 collider 控制阻挡
- 碎块状态更新
- 受击时碎片反应
- 彻底销毁后写入 `WorldState`
- 可能恢复玩家资源

**关卡意义**：

- 这是最直接的“战斗行为改写地形”

**Ark 可借鉴点**：

- 若示巴星后续要做可破坏晶体壁、音叉树根障碍、黑水凝结壳，这一类非常值得借

##### 14. `CrystalDestroyable`

**本质**：资源矿/可击碎晶体

**特殊点**：

- 可掉 XP
- 和普通障碍物共享 Destroyable 语法

**关卡意义**：

- 用资源可破坏物给探索提供低强度正反馈

**Ark 可借鉴点**：

- 可作为小型资源物、逻辑图谱/废料/记忆碎片承载物的参考

##### 15. `PlantDestroyable`

**本质**：轻量级、可再生、低阻挡环境物

**特殊能力**：

- 可触碰即碎
- 可生长回来
- 可恢复能量
- 可阻止敌人破坏

**关卡意义**：

- 这类东西不是核心机关，而是“让世界有可玩手感”的柔性资产

**Ark 可借鉴点**：

- 示巴星中的克拉德尼花、声学植被、脆弱晶丛，适合做成这类“轻反馈环境物”

---

#### E. 传送 / 强制位移 / 空间转送部件

##### 16. `LocationManager.GoToLocation(...)`

**本质**：Minishoot 一切正规空间切换的总入口

**做的事**：

- 停止镜头 stick
- 重置视觉状态
- 按目标入口方向重定位玩家
- 激活目标 Location
- 根据情况播放落洞/过门等转场

**关卡意义**：

- 它说明 Minishoot 的空间切换是一个统一流程，而不是每个门自己传送玩家

**Ark 可借鉴点**：

- 房间切换逻辑最好也集中在 `RoomManager / TransitionCoordinator` 一类统一入口中

##### 17. `TeleportOutTrigger`

**本质**：特殊剧情型强制转送器

**触发方式**：

- 玩家进入 trigger
- 满足某个世界状态后触发

**做的事**：

- 白屏
- 短等待
- 暂停玩家
- 强制把玩家转送到指定入口（`Overworld > Snow_0`）

**是否持久化**：

- 自身不直接持久化，但受 `WorldState` 条件控制

**关卡意义**：

- 说明 Minishoot 除了标准门切换外，也允许少量“剧情式非对称传送”

**Ark 可借鉴点**：

- 你未来完全可以有少量高仪式感的非标准转送，如坠落、共鸣牵引、黑水吞没、索尔基音拖拽

---

### 1.28 一个总总结：Minishoot 的互动件不是“机关库”，而是“状态改写库”

看完上面的部件，会发现 Minishoot 的互动件虽然种类不少，但底层其实很统一。  
它们几乎都在做同一件事：

> **把玩家当前行为，翻译成某个空间状态的改变。**

这些状态改写大致分为 4 类：

1. **结构改写**
   - 门开了
   - 路通了
   - 障碍碎了

2. **可见性改写**
   - 隐藏区淡出
   - 埋伏显形
   - 新入口被看见

3. **节奏改写**
   - 进入清算战
   - 触发波次
   - 镜头 travel 到目标

4. **持久性改写**
   - 某个节点以后永远变了
   - 某个钥匙以后一直持有
   - 某个机关已经完成

这给 Ark 的最强启示不是“去复刻火炬或门”，而是：

**任何你想放进示巴星的互动件，都应该先问：它改写的到底是哪一种空间状态？**

如果这个问题答不出来，那它多半只是装饰性 gimmick，不是高质量关卡部件。

### 1.29 Minishoot 代表性房间样本拆解

> 说明：以下不是对每一个具体房间的逐像素复原，而是基于 **场景对象命名、节点邻接关系、脚本职责、触发器语义** 做出的高可信模式化重建。  
> 目的不是“考古截图”，而是提炼出可直接移植到 `Project Ark` 的房间样板。

---

#### 样本 1：`Cave` 开场引导房

**证据来源**：

- `Cave.unity` 内存在 `CameraTriggerIntro`
- 同场景首个 `Checkpoint` 为 `CaveCheckpoint0`
- `Cave` 内部存在大量 `EncounterOpen`，而不是一开始就密集 `EncounterClose`

**推定房间职责**：

- 接管玩家控制
- 建立基础移动与阅读方向
- 用镜头告诉玩家“你正在进入第一个正式段落”

**房间组成**：

- 一个偏温和的空间入口
- `CameraTriggerIntro` 收束镜头
- 极少量或无锁门战
- 可能有 1 个低风险开放敌群
- 一个非常早期的明确 Checkpoint

**节奏功能**：

- 让玩家从“进入游戏”过渡到“开始读关卡”
- 不是考核房，而是校准房

**为什么有效**：

- 它先建立感知坐标系，再开始施压
- 镜头触发器让空间段落感先于复杂机制出现

**翻译成 Ark 的做法**：

- 示巴星切片第一个房不要急着上核心机制考核
- 先给：
  - 一个强地标
  - 一个镜头收束
  - 一点轻敌压
  - 一次“我知道该往哪去”的确认

**Ark 房间原型名建议**：

- `SH-SLICE-R01 坠机苏醒房`

---

#### 样本 2：`Cave` 首个“结构句号”房

**证据来源**：

- `CaveCheckpoint0` 与多个 `Transition` 邻接
- 其附近至少有 `Cave > Overworld_26`、`Cave > Cave_4`、`Cave > Cave_5`

**推定房间职责**：

- 作为 Cave 内部的早期结构锚点
- 给出“往外走 / 继续深入 / 回返”的初级结构认知

**房间组成**：

- 一个 Checkpoint
- 至少两个以上出口/连接
- 附近敌压相对降低
- 可能是短回报段或缓冲段

**节奏功能**：

- 让玩家在第一次紧张探索后有一个“重新整理脑图”的地方
- 告诉玩家：这个场景不是线性走廊，而是有内部结构的

**为什么有效**：

- Checkpoint 与多出口相邻，会自然让玩家把这里记成“节点”
- 这会增强回返时的方向感

**翻译成 Ark 的做法**：

- 第一个安全点不要只放在直线尽头
- 尽量让它连接至少两个语义不同的出口：
  - 主推进出口
  - 回返/捷径出口
  - 未来能力门出口

**Ark 房间原型名建议**：

- `SH-SLICE-R07 第一个安全节点`

---

#### 样本 3：`Cave` 主题锚点房（`Museum` / `Primordial` 型）

**证据来源**：

- `Cave.unity` 存在明确命名的：
  - `CameraTriggerPrimordial`
  - `CameraTriggerMuseum`
  - `CameraTriggerAcedemyRuin`

**推定房间职责**：

- 在同一大场景中，切出具有强主题身份的子章节
- 通过命名型镜头点，把“普通通路”变成“值得记住的房”

**房间组成**：

- 一个专用 `CameraTrigger`
- 很可能对应独特景观、叙事物件、Boss 前兆、特殊遗迹或能力相关内容
- 不一定依赖大规模战斗

**节奏功能**：

- 在战斗与探索之间插入高辨识度的“叙事锚点”
- 让玩家脑中形成：哦，这里是那个 Museum / Primordial 的地方

**为什么有效**：

- 玩家记住房间，不是靠坐标，而是靠身份
- 命名型触发器意味着设计师非常明确地区分“普通空间”和“章节空间”

**翻译成 Ark 的做法**：

- 示巴星前段至少需要 1 个强身份锚点房
- 比如：
  - `索尔冰雕房`
  - `花路回声房`
  - `音叉林低语房`

它们可以不是大战斗房，但必须让玩家记得住

**Ark 房间原型名建议**：

- `SH-SLICE-R03 世界锚点房`

---

#### 样本 4：`Overworld` 开放压力房

**证据来源**：

- `Overworld.unity` 中 `EncounterOpen` 数量远高于 `EncounterClose`
- 比例大约是 `70+ : 16`

**推定房间职责**：

- 维持开放世界中的持续敌意
- 让玩家始终处于“边探索边处理压力”的状态
- 但不强制每次停下来打完

**房间组成**：

- 开放敌群
- 可能配合简单地形差异
- 没有封门
- 玩家可绕、可拉扯、可离开视野后脱战

**节奏功能**：

- 形成世界底噪
- 让推进路上一直有活性，而不是纯空旷赶路

**为什么有效**：

- 玩家保持戒备，但不会因为频繁封门而疲劳
- 整个世界显得在呼吸，而不是由离散房间拼起来

**翻译成 Ark 的做法**：

- 示巴星前段需要 1-2 个“开放骚扰房”
- 它们的目标不是挑战玩家，而是让玩家习惯：
  - 噪音会引来东西
  - 静默并不安全
  - 战斗可以是局部处理，不一定每次都清场

**Ark 房间原型名建议**：

- `SH-SLICE-R02 静默压力房`

---

#### 样本 5：`OverworldEncounterClose0` 型的节点清算房

**证据来源**：

- `OverworldEncounterClose0` 附近可见敌对对象，如 `Cannon`
- 与大量开放敌群并存，但它自己被命名为单独的 `EncounterClose`

**推定房间职责**：

- 在开放大世界中放置“必须正面解决”的关键节点
- 常用于：
  - 路口清障
  - 关键资源保护
  - 教学考核
  - 小型据点战

**房间组成**：

- 触发器
- 围栏/封场
- 一到多波敌人，可能有固定炮台
- 战后解锁/掉落/清出通路

**节奏功能**：

- 把开放流中的局部“逗号”变成“句号”
- 告诉玩家：这里不是骚扰，这是结构性阻挡

**为什么有效**：

- 开放区域因为少量高权重清算点而更有层次
- 玩家能分清“这次不用清”和“这次必须清”

**翻译成 Ark 的做法**：

- 示巴星切片需要至少 1 个这种房
- 而且它最好承担“验证核心 loop”的职责：
  - 战斗 = 散热窗口
  - 封场 = 逼你真正使用这个理解

**Ark 房间原型名建议**：

- `SH-SLICE-R04 第一清算房`

---

#### 样本 6：`OverworldCheckpoint0 + Dungeon1~5 入口群` 型导航枢纽

**证据来源**：

- `OverworldCheckpoint0` 附近能看到：
  - `Overworld > Dungeon1_0`
  - `Overworld > Dungeon2_0`
  - `Overworld > Dungeon3_0`
  - `Overworld > Dungeon4_0`
  - `Overworld > Dungeon5_0`
  - 以及若干返回 `Cave` / `Tower` 的入口

**推定房间职责**：

- 作为世界导航平面上的高密度分流节点
- 给玩家一个“战略选择”的空间，而非单一路径推进

**房间组成**：

- 一个核心 Checkpoint
- 多个高价值入口
- 极强的脑图锚定价值
- 通常压力相对可控，不会是最激烈战斗区

**节奏功能**：

- 让玩家在一个熟悉节点上反复重读世界结构
- 是世界地图感形成的关键

**为什么有效**：

- 玩家一旦记住这个节点，后续所有新入口都会被挂在一个稳定的空间心智模型上

**翻译成 Ark 的做法**：

- 示巴星完整版本以后，也一定需要至少一个“导航枢纽式房间”
- 哪怕切片阶段做不到完整枢纽，也要开始保留这种思路：
  - Safe Room 不只是恢复点
  - 它应逐渐承担世界分流中心的角色

**Ark 房间原型名建议**：

- 长期可扩展为 `Sheba Central Hub` 或 `回音大教堂前厅`

---

#### 样本 7：`DoorBossDungeon / DoorBossDungeon2` 型章节门槛房

**证据来源**：

- `Overworld.unity` 中存在 `DoorBossDungeon`、`DoorBossDungeon2`
- 这类命名与 Dungeon 入口强相关，明显不是普通门

**推定房间职责**：

- 作为大章节门槛
- 在“我看到入口”与“我有资格进去”之间制造结构差

**房间组成**：

- 入口可见
- 门槛明确
- 常与钥匙、Boss 完成度、主线推进绑定

**节奏功能**：

- 给玩家长期目标
- 形成“看得见但暂时进不去”的银河城承诺

**为什么有效**：

- 它能把好奇心转化成长期动机
- 同时避免世界早期完全摊平

**翻译成 Ark 的做法**：

- 示巴星早期也应该有至少一个“可见但暂不可达”的承诺点
- 比如远处哭泣钟楼、某个黑水封锁区、某个需要 `涟漪` 才能进入的深层入口

**Ark 房间原型名建议**：

- `SH-SLICE-R05 远景承诺房` 或 `后续能力门预告房`

---

#### 样本 8：`HiddenArea + EncounterHidden` 型隐性奖励/惩罚房

**证据来源**：

- 存在 `HiddenArea`
- 存在 `EncounterHidden`
- 二者都接入激活/反激活与 `WorldState` 逻辑

**推定房间职责**：

- 奖励玩家对空间细节的注意
- 在开放探索里插入“非主线的可选惊喜”

**房间组成**：

- 一个视觉遮挡层
- 背后可能藏：
  - 资源
  - 小敌群
  - 伏击
  - 稀有路径

**节奏功能**：

- 让探索不只是走主路
- 给细读地图的玩家额外反馈

**为什么有效**：

- 它增强了“世界不是为你摊开的”的感觉
- 也让每次回返有重新检查旧区域的价值

**翻译成 Ark 的做法**：

- 示巴星非常适合做这种房：
  - 花幕背后
  - 音叉林阴影里
  - 黑水低语裂缝后
  - 缕拉经过留下的异常共鸣点

**Ark 房间原型名建议**：

- `SH-SLICE-R06 可选隐藏支房`

### 1.31 最后落回 Ark：示巴星首切片应该如何组合这些模板

如果只做一个最小可玩切片，我建议直接按下面顺序组合：

引导锚点房

这套顺序为什么有效：

- 有开场气质
- 有世界压力
- 有记忆点
- 有正式证明
- 有回报
- 有收束

这正是 Minishoot 最值得借的地方：

**不是某个房间很神，而是房间模板之间的组合顺序很稳。**

### 1.32 把 8 类样本继续下沉：每类房间的标准组件清单 + 进出规则 + 奖励规则

下面这一节的目标很明确：

- 不是再解释“这种房间是什么”
- 而是定义“这种房间落地时最低限度该放什么”

也就是说，这一节更接近 **关卡生产模板**，而不是分析散文。

---

#### 样本 1：开场引导房

**标准组件清单**：

- 1 个入口出生点或落点
- 1 个显性视线地标
- 1 个 `CameraTriggerIntro` 型镜头收束器
- 0-1 组低威胁 `EncounterOpen`
- 1 条绝对明确的主前进路径
- 0-1 个很轻的教学互动件

**进出规则**：

- 进入条件：
  - 玩家刚进入区域
  - 不要求前置钥匙/能力
- 退出条件：
  - 玩家理解主方向
  - 玩家完成最基础的移动/观察
  - 不宜设置封门或高失败惩罚

**奖励规则**：

- 奖励以“信息确认”优先，而不是物资优先
- 最适合给：
  - 地标显现
  - 第一条叙事线索
  - 第一次世界机制预告
- 不建议给大资源，因为玩家此时还没有形成资源价值判断

---

#### 样本 2：结构锚点房（Checkpoint 邻接型）

**标准组件清单**：

- 1 个 `Checkpoint`
- 2-3 个语义不同的出口
- 1 个低噪音安全区
- 0-1 个轻量回报物
- 1 个足够易记的空间轮廓或视觉符号

**进出规则**：

- 进入条件：
  - 通常位于一段探索或一场小考核之后
- 退出条件：
  - 玩家可以从这里做路线选择
  - 至少一个出口是当前主路
  - 至少一个出口承担未来回返/捷径/能力门语义

**奖励规则**：

- 奖励以“结构收益”优先
- 最适合给：
  - 存档/恢复
  - 地图认知
  - 快捷回返点
- 可附带少量补给，但不应把重心放在掉宝上

---

#### 样本 3：主题锚点房（Museum / Primordial 型）

**标准组件清单**：

- 1 个专用命名镜头触发器
- 1 个强身份景观或叙事物件
- 0-1 个独特互动件
- 0-1 个轻战斗或完全无战斗
- 1 段足以让玩家停下来的观察窗口

**进出规则**：

- 进入条件：
  - 玩家已经经历过 1 段常规流，准备接受章节身份强化
- 退出条件：
  - 玩家获得明确记忆标签
  - 房间出口最好导向下一个高权重节点，而不是立刻回到纯过路房

**奖励规则**：

- 奖励以“身份确认”优先
- 最适合给：
  - 新区域命名感
  - NPC 痕迹
  - 特殊视觉/声音体验
  - 对后续机制的上下文解释
- 如果给物资，最好是纪念性或稀有性，而不是常规消耗品

---

#### 样本 4：开放压力房

**标准组件清单**：

- 1 组 `EncounterOpen`
- 1-2 个能制造走位差的地形元素
- 1 条可绕行但不完全无脑的路径
- 1 个离场边界或脱战可能性
- 0-1 个轻隐藏点

**进出规则**：

- 进入条件：
  - 常位于两个高权重节点之间
  - 用来维持世界底噪
- 退出条件：
  - 玩家不必清空全部敌人
  - 只要能安全通过或局部处理压力即可离开
  - 不应把主路线锁死在“清完所有敌人”上

**奖励规则**：

- 奖励以“低强度即时收益”优先
- 最适合给：
  - 少量资源
  - 小幅补血/补能
  - 一个可被发现的观察细节
- 不宜放大件奖励，否则玩家会被迫把每个开放房都当成清算房

---

#### 样本 5：节点清算房（EncounterClose 型）

**标准组件清单**：

- 1 个 `EncounterClose`
- 封门或封边界组件
- 1-3 组清晰分层的敌人组合
- 1 个战后解锁反馈
- 1 个安全出口或回报出口

**进出规则**：

- 进入条件：
  - 玩家接近结构性阻挡点
  - 通常应先看见门槛，再踩入触发区
- 退出条件：
  - 必须完成敌群清除/波次完成
  - 战后明确解锁：
    - 门开
    - 路通
    - 奖励出现

**奖励规则**：

- 奖励以“推进许可”优先
- 最适合给：
  - 新通路
  - 钥匙
  - 核心资源包
  - 一个短暂安全窗口
- 这类房最重要的奖励不是 loot，而是“你被承认为通过”

---

#### 样本 6：导航枢纽房（OverworldCheckpoint + 多入口型）

**标准组件清单**：

- 1 个高辨识度 `Checkpoint`
- 3 个以上方向明确的入口/出口
- 1 个中央安全活动区
- 1 个清楚可读的远景或地图锚点
- 0-1 个服务型互动件

**进出规则**：

- 进入条件：
  - 玩家完成一个子目标后回到这里，或从这里发散到多个目标
- 退出条件：
  - 玩家需要做战略选择，而不是只沿单线继续前进
  - 各出口应当有不同语义：
    - 主线
    - 支线
    - 回返
    - 暂不可达

**奖励规则**：

- 奖励以“导航权”优先
- 最适合给：
  - 已解锁路标
  - 地图更新
  - 快捷通道
  - 区域再进入便利性
- 不适合做成纯物资仓库，否则会削弱它的战略功能

---

#### 样本 7：章节门槛房（Boss Door / Dungeon Door 型）

**标准组件清单**：

- 1 个高可见度门槛对象
- 1 个门锁条件载体
- 1 个远景或前景构图，用来强调“我看得见它”
- 0-1 个前哨战或守门敌
- 1 个失败后仍可读懂的反馈

**进出规则**：

- 进入条件：
  - 玩家先看见，再知道进不去，最后在未来满足条件后回来
- 退出条件：
  - 如果未满足条件，只能原路离开或转向别的路线
  - 如果满足条件，进入应当具有仪式感，不应像普通门一样轻飘飘

**奖励规则**：

- 奖励以“长期承诺”优先
- 最适合给：
  - 未来目标
  - 剧情预告
  - 大章节入口感
- 这类房的即时奖励可以很少，因为它本身就是动机投放器

---

#### 样本 8：隐藏支房（HiddenArea / EncounterHidden 型）

**标准组件清单**：

- 1 个视觉或空间遮挡层
- 1 个可被细读发现的异常提示
- 0-1 个隐藏敌群或伏击
- 1 个小奖励点
- 1 条短路径或小口袋空间

**进出规则**：

- 进入条件：
  - 玩家主动偏离主路
  - 或观察到异常后尝试验证
- 退出条件：
  - 不影响主线推进闭环
  - 最好可快速回到主路，避免隐藏支路过长打断主节奏

**奖励规则**：

- 奖励以“细读回报”优先
- 最适合给：
  - 稀有资源
  - 叙事碎片
  - 小型捷径
  - 独特视觉发现
- 如果放伏击，奖励必须明显高于普通过路房，否则玩家会学会“不值得探索”

---

### 1.33 再压一层：8 类样本各自最适合承担什么奖励

如果只想快速判断“某个奖励应该塞进哪类房间”，可以直接按下面的对照：

| 房间类型 | 最适合的奖励 | 不适合的奖励 |
| --- | --- | --- |
| 开场引导房 | 信息确认、世界观预告、方向确认 | 大量资源、强考核 |
| 结构锚点房 | 存档、恢复、路线认知、捷径 | 高压封门战 |
| 主题锚点房 | 叙事身份、章节记忆、稀有观察物 | 常规掉宝堆 |
| 开放压力房 | 少量即时资源、轻隐藏、局部战利品 | 关键主线路径许可 |
| 节点清算房 | 推进许可、核心资源包、门锁解除 | 纯装饰性奖励 |
| 导航枢纽房 | 地图更新、分流权限、服务功能 | 重复性消耗品堆放 |
| 章节门槛房 | 长期目标、未来承诺、章节入口感 | 轻飘的普通奖励 |
| 隐藏支房 | 稀有资源、秘密叙事、小捷径 | 无意义空房或低收益伏击 |

### 1.34 对 Ark 最有用的一条生产纪律

做示巴星时，可以把每个房间都强制回答这 3 个问题：

1. 这个房间的**最低标准组件**是什么？
2. 玩家**进入**它靠什么条件，**离开**它靠什么条件？
3. 它发的奖励到底是：
   - 信息奖励
   - 结构奖励
   - 推进奖励
   - 资源奖励
   - 长期承诺奖励

如果这 3 个问题答不清，这个房间大概率还只是“概念房”，不是“可生产房”。

### 1.35 示巴星房间生产检查表

下面这张表的用途不是分析 Minishoot，而是服务 `Project Ark` 的实际搭建。

使用方式很简单：

- 先给房间定类型
- 再逐列检查
- 任意一列答不上来，这个房间就先不要进制作

---

#### A. 通用总检查表

每个示巴星房间，无论类型，都先过这一轮：

| 检查项 | 必问问题 | 合格标准 |
| --- | --- | --- |
| 房间身份 | 这个房间属于哪一类？ | 必须能明确归入 8 类之一，不能是“什么都想做” |
| 主职责 | 它主要提供什么？ | 只能有 1 个主职责，最多 1 个副职责 |
| 主路径 | 玩家一眼能不能知道主方向？ | 至少有一个清晰主前进方向 |
| 入口条件 | 玩家为什么能进入这里？ | 入口条件明确，不靠偶然误入 |
| 退出条件 | 玩家凭什么离开这里？ | 退出条件明确：通过 / 清场 / 解锁 / 发现 / 回返 |
| 空间记忆点 | 玩家离开后会怎么记住它？ | 至少有 1 个地标、构图或机制记忆点 |
| 奖励类型 | 这个房间到底发什么奖励？ | 必须明确属于信息 / 结构 / 推进 / 资源 / 长期承诺之一 |
| 节奏位置 | 它处在切片节奏的哪一拍？ | 明确位于开场 / 压力 / 证明 / 回报 / 收束中的一个位置 |
| 可复用标准件 | 这个房间依赖哪些标准件？ | 尽量用已有房间标准件，而不是临时发明专属脚本 |
| 生产边界 | 这个房间做 MVP 时最少要保留什么？ | 必须能缩成一个最小可玩版本 |

---

#### B. 8 类房间专项检查表

| 房间类型 | 必备组件 | 入口必须满足 | 退出必须满足 | 首选奖励 | 生产时最常见错误 |
| --- | --- | --- | --- | --- | --- |
| 开场引导房 | 出生点、主地标、镜头收束、明确主路 | 玩家初次进入区域，无前置门槛 | 玩家已理解方向与基本观察逻辑 | 信息奖励 | 一上来就塞战斗、叙事、机关、掉宝，导致房间失焦 |
| 结构锚点房 | Checkpoint、多出口、安全空地、易记轮廓 | 玩家刚完成一小段探索或考核 | 玩家可从此做路线选择 | 结构奖励 | 做成单出口休息站，结果没有“锚点”作用 |
| 主题锚点房 | 命名镜头、强身份景观、观察窗口、可选轻互动 | 玩家已经进入该章节语境 | 玩家获得明确章节记忆标签 | 信息奖励 / 稀有观察奖励 | 只有美术气氛，没有结构作用，离开后记不住 |
| 开放压力房 | 开放敌群、走位地形、可绕行路径、脱战空间 | 位于高权重节点之间，用来维持底噪 | 玩家可通过、绕开或局部处理后离场 | 低强度资源奖励 | 把它做成变相清算房，逼玩家每次都清空 |
| 节点清算房 | EncounterClose、封边界、分层敌群、战后解锁反馈 | 玩家接近一个明确的结构阻挡点 | 必须清场或完成波次后才能走 | 推进奖励 | 清完只掉一点杂物，没有结构回报，导致考核不值 |
| 导航枢纽房 | 高辨识度 Checkpoint、多方向出口、中央安全区、地图锚点 | 玩家准备发散到多个目标，或完成支线后回流 | 玩家必须能做真正的路线决策 | 结构奖励 / 导航奖励 | 把所有出口都做成同语义通道，导致枢纽不成枢纽 |
| 章节门槛房 | 高可见门槛、门锁条件、远景构图、失败反馈 | 玩家先看到，再知道暂时进不去 | 未满足条件时能读懂阻挡；满足后进入有仪式感 | 长期承诺奖励 | 门只是一个普通锁，既没有承诺感也没有章节感 |
| 隐藏支房 | 遮挡层、异常提示、小口袋空间、小奖励点 | 玩家主动偏离主路或读出异常 | 可快速回主路，不打断整体节奏 | 稀有资源 / 秘密叙事 | 隐藏支路太长、太空、奖励太弱，玩家下次不想找 |

---

#### C. 示巴星首切片专用简化版

如果只服务于示巴星第一个可玩切片，我建议把检查再压成下面这一版。  
每做一个 `SH-SLICE-Rxx`，至少要写满这 7 项：

| 字段 | 要填写的内容 |
| --- | --- |
| 房间 ID | 例如 `SH-SLICE-R03` |
| 房间类型 | 8 类之一 |
| 主职责 | 例如“世界锚点”或“第一清算” |
| 必备组件 | 最少 3-5 个，不许写空话 |
| 进入条件 | 玩家为什么现在会来到这里 |
| 退出条件 | 玩家凭什么能离开这里 |
| 奖励结算 | 玩家从这里实际得到什么 |

---

#### D. 一个建议的填写范例

```text
房间 ID：SH-SLICE-R04
房间类型：节点清算房
主职责：验证“噪音会升温，升温会逼你移动和控场”的核心 loop
必备组件：封场边界、2 波敌群、1 个安全柱体、战后开启的门、战后掉落的冷却资源
进入条件：玩家已见过开放压力房，并已学会最基础移动射击
退出条件：清空两波敌人，门打开
奖励结算：主路线推进许可 + 一次明确的资源回补
```

---

#### E. 最后的执行纪律

这张检查表最重要的不是“写得完整”，而是强迫每个房间都满足这三个要求：

1. **它有单一主职责**
2. **它有明确通过条件**
3. **它有与风险匹配的奖励**

只要有一个房间在这三点上含糊，整个切片的节奏就会开始发散。

## 2. Silksong

### 2.1 先说结论：Silksong 的关卡骨架看起来更像“房间图谱 + 门名语法 + 区域导航系统”

如果先用一句话概括目前能确认的 `Silksong` 关卡架构，我会写成：

> **SceneTransitionZone 用“目标场景 + 目标 Gate 名”缝合世界，WorldNavigation 把全部房间与出口整理成显式图谱，CameraController 和 AreaTitleController 控制房间进入时的阅读方式，而 GameMap / WideMap / FastTravel 系统负责把复杂路径重新压回玩家可理解的导航结构。**

它和前面两款参考游戏的区别很明显：

- `Minishoot` 的强项是稳定清晰的房间职责语法
- `TUNIC` 的强项是认知重读与知识门槛
- `Silksong` 目前显现出来的强项更像：
  - **大规模房间网络**
  - **高度标准化的出口命名**
  - **区域标题 / 地图 / 快速旅行共同构成的导航闭环**
  - **平台件与机关件穿插在路径推进里**

换句话说，`Silksong` 最值得 Ark 借的，初步看不是单个 gimmick，而是：

**如何把一个庞杂、层叠、门很多的世界，依然组织得可导航、可记忆、可回返。**

---

### 2.2 证据边界说明

这次 `Silksong` 参考资产和 `TUNIC` 不同：

- 目前拿到的完整 `.unity` 场景文件非常少，几乎无法靠场景 YAML 做逐房间还原
- 但拿到了大量 `Assembly-CSharp` 脚本

因此这一轮分析主要基于：

- `SceneTransitionZone`
- `SceneTransitionZoneBase`
- `WorldNavigation`
- `CameraController`
- `AreaTitleController`
- `GameMap`
- `InventoryWideMap`
- `FastTravelMapButton`
- `Trapdoor`
- `TripWire`
- `PositionActivator`
- 以及脚本集中出现的大量 `PlayMaker Actions`

所以现在的结论会更偏：

- 世界组织方式
- 转场与门名语法
- 地图/导航架构
- 平台与机关标准件

而不是伪装成已经做完逐房间考古。

---

### 2.3 第一层：Silksong 的世界是显式的“场景图”，不是隐式拼起来的连续地图

`WorldNavigation` 是这次最关键的入口之一。

它直接定义了：

- `SceneItem`
  - `Path`
  - `Name`
  - `Transitions`
- `TransitionItem`
  - `Name`

更重要的是，它内置了一个非常大的 `Scenes` 表，当前可见数量是：

- `SceneItem[] Scenes = new SceneItem[376]`

并且每个场景都带一组显式出口名，例如：

- `Town`
  - `right1`
  - `door_dreamReturn`
  - `top1`
  - `left1`
  - `door_bretta`
  - `door_jiji`
  - `door_sly`
  - `door_station`
  - `door_mapper`
  - `bot1`
  - `room_divine`
  - `room_grimm`

这件事的意义非常大：

`Silksong` 的世界不是靠“设计师心里知道这些房间怎么连”，而是已经被整理成一张显式的 **场景-出口图谱**。

这意味着它的关卡生产天然有几个优势：

- 所有房间都有明确身份
- 所有出口都有明确名字
- 世界图可以被工具化、可视化、校验化
- 后续地图和快速旅行系统更容易复用同一份结构数据

这点和 Minishoot 很像，但规模感更强，也更接近传统银河城“多房间网络”的生产方式。

---

### 2.4 第二层：转场不是模糊的 trigger，而是“目标场景 + 目标 Gate”的双字段语法

`SceneTransitionZone` 很简单，但非常说明问题：

- `targetScene`
- `targetGate`

而真正逻辑在 `SceneTransitionZoneBase`：

- 进入时检查玩家状态
- `ScenePreloader.SpawnPreloader(TargetScene, LoadSceneMode.Additive)`
- 可配置：
  - `fadeColor`
  - `fadeDuration`
  - `holdDuration`
  - `freezeCamera`
  - 多组音频 snapshot
- 最终调用：
  - `GameManager.instance.BeginSceneTransition(...)`
  - `EntryGateName = TargetGate`
  - `SceneName = TargetScene`

这说明 `Silksong` 的转场是非常标准化的：

**每一个出口都同时回答两个问题：去哪里？从那边的哪个门进？**

这套语法的好处是：

- 关卡逻辑非常清楚
- 回返路径很好维护
- 区域标题、镜头、地图标记都可以基于 entry gate 再做分支

对 Ark 的启示非常直接：

- 未来 `RoomTransition` 最好不要只是“目标房间引用”
- 而要保留：
  - 目标房间
  - 目标入口点 ID
  - 进入该入口时的镜头/标题/触发条件

---

### 2.5 第三层：Silksong 的导航能力，不只靠地图，而是靠“入口门名 + 区域标题 + 地图状态”三者联动

#### A. 区域标题不是装饰，而是入口条件化的章节确认

`AreaTitleController` 里可以确认这些信息：

- `orderedAreas`
- `doorTrigger`
- `doorException`
- `onlyOnRevisit`
- `onlyIfUnvisited`
- `waitForTrigger`
- `triggerEnterWaitForHeroInPosition`
- `VisitedBool`

这说明区域标题的播放不是“走进来就播”这么粗糙，而是会根据：

- 你是不是第一次来
- 你是不是从特定门进来
- 你是不是已经站稳
- 当前满足哪一条 area 条件

来决定是否显示、显示哪一个标题、以什么形式显示。

也就是说，在 `Silksong` 里，**入口门本身会影响房间的章节阅读方式。**

#### B. 地图系统并不是单一贴图，而是按区域/父节点/条件组装出来的

`GameMap` 的结构非常重：

- `ParentInfo`
  - `Parent`
  - `PlayerDataBool`
  - `PositionConditions`
  - `Maps`
- `ZoneInfo`
  - `Parents`
  - `WideMapZoomPositionsOrdered`
  - `QuickMapPositionsOrdered`
  - `LocalBounds`
  - `VisibleLocalBounds`
  - `NameOverride`

而且 `ParentInfo` 还会检查：

- `IsUnlocked`
- `PositionConditions`
- `CheckActivation()`

这说明地图并不是“整张图一次性给玩家”，而是：

- 按区域父节点组织
- 按玩家状态解锁
- 按位置条件改变呈现
- 按局部 bounds 管理显示

#### C. 宽地图和快速地图是分层设计，不是一个 UI 两种缩放

`InventoryWideMap` 又补了一层：

- `positionsOrdered`
- `compassIcon`
- `corpseIcon`
- `InventoryItemWideMapZone[] ZoneConditions`
- `GetCompassPositionLocalBounds()`
- `GetCorpsePositionLocalBounds()`

这说明它不是简单把大地图缩小，而是专门为“wide map”做了另一层导航逻辑，包括：

- 宽图定位偏移
- 指南针图标
- 尸体位置
- 基于 zone 的图标投射

这套设计很像在说：

**地图不是一个画面，而是一整套导航解释器。**

---

### 2.6 第四层：Silksong 看起来非常重视“复杂路径下的可回返性”

从目前读到的脚本组合来看，它显然不只是做“从左到右”的房间链，而是在主动支撑复杂回返。

可确认的证据包括：

- `WorldNavigation` 中每个场景都有多出口
- `FastTravelMapButton`
- `FastTravelMapButtonBase`
- `TubeTravelMapButton`
- `OpenFastTravelUI`
- `GetFastTravelScene`
- `FindFastTravelTeleportPoint`

这说明快速旅行不是外挂式菜单，而是已经深度嵌进房间网络和地图系统里。

换句话说，`Silksong` 很可能有一种很成熟的设计思路：

1. 先让玩家在复杂世界里建立门与区域的关系
2. 再逐步把这些关系压缩成更高阶的回返方式
3. 让庞大世界保持探索感，同时不会因为往返成本过高而崩掉

这点对 Ark 特别重要，因为你后面如果真的把示巴星做成立体多环路地图，**回返结构的质量会直接决定玩家是否觉得“复杂但好玩”还是“复杂且烦”。**

---

### 2.7 第五层：Silksong 的关卡标准件里，平台件和轻机关件明显占很高比重

这部分目前最直接能确认的一批件有：

- `Trapdoor`
- `TripWire`
- `GateSnap`
- `PositionActivator`
- `PositionActivatorRange`
- `SetPosConditional`

这批脚本非常关键，因为它们说明 `Silksong` 的房间并不只是靠敌人和门来组织。

#### A. `Trapdoor`

`Trapdoor` 的字段非常像标准化门机关：

- `doorAnimator`
- `mechanismAnimator`
- `cannotCloseTrigger`
- `openWaitTime`
- `closeWaitTime`
- `PersistentBoolItem`
- `stayOpen`
- `enterSceneTrigger`
- `OnOpen / OnClose / OnClosed / OnStartOpen`

这说明 trapdoor 不是一次性演出件，而是：

- 可持久化
- 可按进场条件设初始状态
- 可受 lever / trigger 控制
- 有完整开闭事件的标准机关件

#### B. `TripWire`

`TripWire` 则更像“轻量节奏机关”：

- trigger
- flash
- activateEffects
- `OnTriggered`

这意味着它的职责可能更多是：

- 提醒玩家
- 改变当前节拍
- 触发小事件
- 让路径不总是静态可走

#### C. `PositionActivator`

`PositionActivator` 特别有意思，它不是“撞到就触发”，而是基于相对位置：

- `relativeTo`
- `positionY`
- `IsNowAbove`
- `IsJustAbove`
- `IsNowNotAbove`
- `IsJustNotAbove`

这说明 `Silksong` 的一些房间逻辑会直接依赖：

- 玩家在某个物体之上还是之下
- 玩家是否越过某条垂直阈值

这非常符合平台银河城的思路，因为它能让设计师把“高低关系”直接写进触发逻辑里。

---

### 2.8 第六层：镜头不是全局跟随器，而是节奏控制器

`CameraController` 透露出来的信息很多，而且都很关卡导向：

- `CameraMode`
  - `FROZEN`
  - `FOLLOWING`
  - `LOCKED`
  - `PANNING`
  - `FADEOUT`
  - `FADEIN`
- `CameraLockArea`
- `lockZoneList`
- `sceneWidth / sceneHeight / xLimit / yLimit`
- `AllowExitingSceneBounds`
- `PositionedAtHero`
- `FreezeInPlace`

这意味着：

- 镜头不只是跟着玩家跑
- 它会：
  - 锁定
  - 平移
  - 冻结
  - 限制在房间边界内
  - 根据房间进入重新定位

对 `Silksong` 这种偏平台化、垂直性强、路径密集的世界来说，这非常重要。  
因为镜头本身就在帮忙完成：

- 读图
- 预告
- 压缩视野
- 释放视野

这点和 `TUNIC` 的“镜头解释空间”不一样。  
`TUNIC` 更像“镜头帮助误导”，而 `Silksong` 更像“镜头帮助平台与路径阅读”。

---

### 2.9 现阶段可以先下的 5 个高可信判断

#### A. Silksong 的底层生产单位，极可能是“独立 scene 房间”

`WorldNavigation` 里直接列出大量 `Assets/Scenes/...`，且每个房间都有出口名。  
这很像 Hollow Knight 系列延续下来的房间式场景组织法。

#### B. 它的入口语法比 Minishoot 还显式

Minishoot 虽然也很清楚，但 `Silksong` 这里直接把所有过门都做成：

- `targetScene`
- `targetGate`

加上一张显式图谱。

#### C. 它非常依赖导航系统来消化复杂性

区域标题、门名、地图、宽图、快速旅行都不是孤立系统，而是同一个导航闭环的不同层。

#### D. 它的房间节奏大概率是“平台推进 + 轻机关切拍 + 战斗节点”交替

目前虽然还没拆到具体房间样本，但从 `Trapdoor`、`TripWire`、`PositionActivator` 这类件可以看出，机关密度应该不会低。

#### E. 它最值得 Ark 借的，可能是“大规模房间网络的可导航生产方法”

这点甚至可能比单个具体房间更重要。

---

### 2.10 对 Ark 的第一批直接启示

#### A. 如果以后房间数量变多，必须尽早建立“入口 ID 语法”

不要等地图大了再补。  
`targetScene + targetGate` 这种双字段结构很值得借。

#### B. 区域标题应该和入口条件绑定，而不是做成一段统一播片

示巴星未来也完全可以根据：

- 从哪个入口进来
- 是否首次到达
- 是否已经理解某个机制

决定标题和播报怎么出现。

#### C. 地图系统最好从一开始就允许“局部解锁”和“条件化显示”

不要默认一张整图。  
`GameMap` 的结构很说明问题：复杂地图必须按区域、条件、节点去组织。

#### D. 房间机关不应该都写成自定义剧情件

`Trapdoor`、`TripWire`、`PositionActivator` 这种件的价值，在于它们是可复用的。

#### E. 镜头边界和锁区最好当成房间标准件

不要把镜头问题留到最后用手调。  
平台银河城里，镜头就是房间语法的一部分。

---

### 2.11 一个总总结：Silksong 现在露出来的，不是“谜题感”，而是“超大房间网络的可控性”

Minishoot 给人的感觉是：

> 房间职责很清楚，世界节奏很稳

TUNIC 给人的感觉是：

> 世界会因为理解变化而被重读

而 `Silksong` 当前从脚本骨架里最明显露出来的是：

> **当房间数量变得非常多、路径关系非常密、区域层级非常深时，如何仍然让玩家能导航、能记忆、能回返。**

这可能正是它对 `Project Ark` 最关键的参考价值。

---

### 2.12 Silksong 机制词典：它如何把大型平台银河城拆成可生产标准件

下面这一节的目标，和前面 `Minishoot` / `TUNIC` 的机制词典一致：

- 不再只停留在总论
- 而是把 `Silksong` 已经露出来的关卡部件，整理成一套可复用语言

从目前脚本证据看，`Silksong` 的标准件非常像一种“高密度房间网络生产工具箱”。

#### A. 世界图谱件：`WorldNavigation`

这是目前最关键的上位结构件。

**可确认职责**：

- 显式记录全部 scene
- 显式记录每个 scene 的 transition name 列表
- 形成可工具化的房间网络数据表

**设计意义**：

- 世界图不再只是设计文档，而是直接被代码持有
- 所有房间连接关系都有统一名字和统一入口

**对 Ark 的映射**：

- 如果以后关卡规模扩大，最好也维护一份正式的 `RoomGraph`
- 不要只靠场景摆放和人脑记忆维护世界结构

#### B. 转场件：`SceneTransitionZone` / `SceneTransitionZoneBase`

这组对象负责最标准的“过门”。

**可确认职责**：

- 指定 `targetScene`
- 指定 `targetGate`
- 可配置：
  - fade
  - hold
  - freezeCamera
  - 音频 snapshot
- 预加载下一场景
- 统一进入 `BeginSceneTransition`

**设计意义**：

- 出口不是泛 trigger，而是正式的“场景边”
- 过门时的视觉、音频、控制状态都被标准化处理

**对 Ark 的映射**：

- 未来 `RoomTransition` 也最好是一个正式标准件，而不是每个门手写切换逻辑

#### C. 区域确认件：`AreaTitleController`

`Silksong` 的区域标题系统明显比普通“播个标题”复杂得多。

**可确认职责**：

- 可按 `orderedAreas` 做条件排序
- 可要求从特定 `doorTrigger` 进入
- 可排除某个 `doorException`
- 可限定：
  - 只在 revisit 时播
  - 只在 unvisited 时播
  - 等待触发后播
  - 等英雄到位后播
- 还会记录 `VisitedBool`

**设计意义**：

- 标题不是 UI 装饰，而是路径解释器
- 同一个区域，从不同门进入，可能被读成不同段落

**对 Ark 的映射**：

- 示巴星的区域播报 / HUD 标题 / 钟声提示，完全可以与入口 ID 和首次到达状态绑定

#### D. 地图分层件：`GameMap` / `GameMapScene`

这是 `Silksong` 导航闭环里最厚的一层。

**1. `GameMap`**

负责：

- 区域父节点管理
- 局部地图对象缓存
- 条件化 zoom / quick map 位置
- bounds 与 visible bounds 管理

**2. `GameMapScene`**

负责：

- 场景映射状态：
  - `Hidden`
  - `Rough`
  - `Full`
- `mappedParent`
- `fullSprite`
- 条件替代 sprite / color
- `hideCondition`

**设计意义**：

- 地图不是“有没有”，而是“映射到什么精度”
- 同一房间可以从未记录 -> 粗略 -> 完整
- 地图可见性还会受额外条件控制

**对 Ark 的映射**：

- 如果以后做 minimap / world map，最好从一开始就考虑：
  - 未发现
  - 已发现但粗略
  - 完整读取
  三段式状态，而不是二元开关

#### E. 宽图导航件：`InventoryWideMap`

这是 `Silksong` 非常实用的一层。

**可确认职责**：

- 不是单纯显示地图
- 会根据条件调整 `PositionOffset`
- 会放置：
  - `compassIcon`
  - `corpseIcon`
- 会根据 zone 找最近节点位置

**设计意义**：

- 宽图不是静态图，而是活导航层
- 玩家在复杂世界里，需要的是“自己现在在哪 + 尸体在哪 + 大方向往哪走”

**对 Ark 的映射**：

- 未来如果做示巴星地图，不要只想着画图
- 还要想：
  - 玩家当前位置怎么解释
  - 关键目标怎么解释
  - 死亡遗留物怎么解释

#### F. 快速旅行件：`FastTravelMapButtonBase` / `FastTravelMapPieceBase` / `FastTravelMapButton` / `TubeTravelMapButton`

这组对象说明快速旅行系统是地图生产链的一部分，而不是外挂菜单。

**可确认职责**：

- 按 `playerDataBool` 判断是否解锁
- 地图按钮与地图上的 piece 成对绑定
- 打开时：
  - 未解锁的按钮隐藏
  - 当前地点自动高亮
- 选中地图项会驱动地图上的 selector 移动

**设计意义**：

- 快速旅行不是抽象列表，而是有空间映射的
- 玩家不是在选一个字符串，而是在选世界里的某个点

**对 Ark 的映射**：

- 未来如果做 fast travel，强烈建议也保留：
  - UI 点
  - 世界图点
  - 当前地点指示
  三者联动

#### G. 镜头锁区件：`CameraController` / `CameraLockArea`

这是 `Silksong` 平台房间读图能力的关键保障。

**可确认职责**：

- `CameraController` 具备：
  - follow
  - lock
  - freeze
  - panning
  - fade
- `CameraLockArea` 具备：
  - `cameraXMin / cameraXMax`
  - `cameraYMin / cameraYMax`
  - `preventLookUp`
  - `preventLookDown`
  - `priority`
  - 基于进入方向设置 `enteredLeft / Right / Top / Bot`

**设计意义**：

- 房间不是只有碰撞边界，还有镜头边界
- 玩家从哪一边进入，会影响镜头如何锁定和解释该房间

**对 Ark 的映射**：

- 未来 `Room` 最好同时拥有：
  - 几何边界
  - 镜头边界
  - 进入方向信息

#### H. 位置阈值件：`PositionActivator` / `PositionActivatorRange` / `SetPosConditional`

这组对象很像 `Silksong` 用来把“平台空间关系”写成可配置逻辑的工具。

**1. `PositionActivator`**

- 判断对象在某个相对高度之上/之下

**2. `PositionActivatorRange`**

- 判断对象是否落在某个相对高度区间内

**3. `SetPosConditional`**

- 按 `PlayerDataTest` 条件切换对象位置

**设计意义**：

- 平台银河城里的“你在上层还是下层”本身就是逻辑条件
- 某些机关、遮挡、路径件不需要复杂脚本，只需要位置条件件组合

**对 Ark 的映射**：

- 如果未来示巴星也要做垂直房间、桥体、坠落井、分层平台
- 非常值得尽早准备这类“位置阈值标准件”

#### I. 轻机关件：`Trapdoor` / `TripWire` / `GateSnap`

**1. `Trapdoor`**

- 持久化
- 开闭事件
- lever / enterSceneTrigger
- 不同方向动画

这是标准的“正式房间机关件”。

**2. `TripWire`**

- 触发
- 闪光
- 抖屏
- 激活效果
- `OnTriggered`

这是标准的“轻节奏切拍件”。

**3. `GateSnap`**

- 看似简单，只做位置 snapping
- 但它说明出口/门对象的摆放本身被标准化了

这点很像生产工具链的一部分，而不是单纯运行时逻辑。

**设计意义**：

- `Silksong` 很可能不是靠大量一次性房间脚本来维持复杂度
- 而是靠很多轻量标准件叠加出复杂房间

#### 1. 世界图谱层

### 2.14 一个总总结：Silksong 的机制词典更像“导航生产工具箱”

目前这轮分析给我的最大印象是：

`Silksong` 的标准件不是围绕“一个房间多神奇”组织的，  
而是围绕“当房间很多、门很多、层次很多时，怎样让整个世界仍然好用”组织的。

这和 `Minishoot`、`TUNIC` 都不一样：

- `Minishoot` 更强调房间职责清晰
- `TUNIC` 更强调认知重读
- `Silksong` 当前更强调：
  - 房间网络可维护
  - 转场语法统一
  - 地图与快速旅行可解释
  - 平台段落可读

这对 `Project Ark` 来说尤其有用，因为你后面只要真正进入“多房间、多区域、多回返”阶段，最先崩的往往不是美术，也不是战斗，而是：

**导航与结构管理。**

而 `Silksong` 看起来正是在帮你回答这个问题。

### 2.15 Silksong 的房间组织法则：它更像“网络节点编排”，不是“单房间奇观编排”

如果把前面的标准件继续往下压，我认为 `Silksong` 当前露出来的房间组织规律可以先总结成 6 条。

#### 法则 1：一个 scene 本身就是正式房间节点，不是随意切分的空间片段

从 `WorldNavigation` 的 `SceneItem[]` 看，`Silksong` 不是只在运行时“碰到出口再说”，而是先有一张显式场景图，再往每个节点里填内容。

这会直接影响房间生产方式：

- 先定义这个房间在世界图里的职责
- 再定义它有哪些入口和出口
- 最后才去决定机关、敌人、奖励和镜头

也就是说，`Silksong` 更像是：

**先做网络节点，再做节点内部戏剧。**

#### 法则 2：入口方向会改变房间的第一阅读版本

`SceneTransitionZoneBase` 明确把 `targetScene + targetGate` 作为转场语法；`AreaTitleController` 又能根据 `doorTrigger / doorException` 决定区域标题是否播放。

这说明在 `Silksong` 里：

- “进入这个房间”不是唯一状态
- “从哪个 gate 进入这个房间”才是第一阅读条件

因此同一房间至少可能有两种职责：

- 首次进入时，它是承诺房
- 从另一边返回时，它变成确认房或压缩房

这和很多平台银河城的实际体验是吻合的：

- 你第一次觉得这个房间大、危险、陌生
- 第二次从另一侧回来，会发现它只是一个结构节点

#### 法则 3：镜头锁区不是附属品，而是房间边界的一部分

从 `CameraLockArea` 看，房间边界至少同时包括：

- 几何边界
- 镜头边界
- 进入方向
- 上下看限制
- 优先级

这意味着 `Silksong` 的“房间”并不只是地形块，而是一个完整阅读单元。

换句话说，平台房间的构成不是：

> 地形 + 敌人

而更接近：

> 地形 + 镜头解释 + 进入朝向 + 路径门槛

#### 法则 4：平台推进靠“轻机关切拍”，而不是每房一个重机关

`TripWire`、`PositionActivator`、`PositionActivatorRange`、`SetPosConditional`、`Trapdoor` 这一组标准件说明：

- 很多节奏变化是轻量的
- 很多路径状态切换是条件化的
- 很多房间的戏剧性来自“空间关系改变”，不是大型谜题

所以 `Silksong` 的房间组织很可能倾向于：

- 主推进动作
- 一个轻门槛
- 一个镜头确认
- 一个短反馈

再继续往下一个节点，而不是在单房间里塞满重逻辑。

#### 法则 5：导航锚点会周期性重置玩家的认知负担

从 `AreaTitleController`、`GameMap`、`InventoryWideMap`、`FastTravelMapButtonBase` 可以看出，`Silksong` 并不是让玩家一直靠裸记忆撑完整个世界。

它会周期性提供：

- 区域命名
- 地图状态更新
- 宽图定位
- 当前地点高亮
- 快速旅行节点

这意味着它的大世界节奏不是一路加压，而是：

**探索扩张 -> 导航确认 -> 再扩张。**

这是一种很成熟的超大地图节奏控制方式。

#### 法则 6：回返路径不是附赠，而是房间图的正式第二形态

`Trapdoor` 的持久开闭、`enterSceneTrigger` 的进场状态、快速旅行点与区域标题系统都说明：

- 某些门第一次是阻挡
- 第二次已经被写成捷径
- 第三次则可能只是路网的一部分

也就是说，很多房间不是只有一个生命周期，而是至少有三种状态：

1. 首次进入状态
2. 解锁后状态
3. 回返压缩状态

这正是大型银河城能够越玩越顺手的关键原因。

---

### 2.16 Silksong 的节奏法则：它更像“长链推进中的周期性减负”

如果把它压成体验节奏，我会这样总结：

#### A. 长推进段

负责：

- 连续平台挑战
- 若干相邻房间的方向判断
- 小风险累积

常见支撑件：

- `SceneTransitionZone`
- `CameraLockArea`
- 位置阈值件

#### B. 轻切拍段

负责：

- 用小机关打断单调推进
- 给玩家一个“这里发生了结构变化”的即时确认

常见支撑件：

- `TripWire`
- `Trapdoor`
- `SetPosConditional`

#### C. 导航确认段

负责：

- 告诉玩家自己到了哪
- 告诉玩家此地在区域结构中的位置
- 暂时降低迷失成本

常见支撑件：

- `AreaTitleController`
- `GameMapScene`
- `InventoryWideMap`

#### D. 回返压缩段

负责：

- 让之前沉重的推进链在回返时变短
- 把一次“征服”变成以后稳定可复用的路径

常见支撑件：

- 持久 `Trapdoor`
- 快速旅行点
- 入口 gate 的另一侧重读

因此 `Silksong` 的大节奏不是：

> 压力 -> 奖励 -> 压力 -> 奖励

而更像：

> 推进 -> 轻切拍 -> 推进 -> 导航确认 -> 再推进 -> 最后把一段路压缩成未来捷径

这个节奏非常适合大图，不容易把玩家直接拖进纯迷路疲劳。

### 2.17 把 Silksong 房间先压成 5 个高可信模板

虽然我们现在还没有逐场景房间考古，但仅凭系统证据，已经足够压出一批高可信模板。

#### 模板 1：通道推进房

**职责**：

- 负责把玩家从一个节点推到下一个节点
- 用镜头锁区和高低差让通道本身具备阅读感

**标准组件**：

- `SceneTransitionZone`
- `CameraLockArea`
- 平台地形
- 1-2 个轻门槛或战斗点

**常见入口 / 出口结构**：

- `left1 / right1`
- `top1 / bot1`

这是大型地图里最基础也最重要的房型。

#### 模板 2：垂直筛选房

**职责**：

- 用高低差筛选当前能力与路径选择
- 让“是否能上去 / 是否能安全落下”变成结构判断

**标准组件**：

- `CameraLockArea`
- `PositionActivator`
- `PositionActivatorRange`
- 单向落差或分层平台

**作用**：

- 首次进入时形成压力
- 回返时形成结构熟悉感

#### 模板 3：轻机关切拍房

**职责**：

- 在长推进链中插入一个短促的结构变化
- 用低成本互动让玩家保持注意力刷新

**标准组件**：

- `TripWire`
- `Trapdoor`
- 触发特效 / 抖屏 / 开门反馈

**作用**：

- 不承担章节级谜题
- 只承担局部节奏重音

#### 模板 4：区域确认房

**职责**：

- 重新解释玩家所处区域
- 让迷路压力暂时下降

**标准组件**：

- `AreaTitleController`
- `GameMapScene`
- 视野较开阔的构图
- 清晰出口组织

**作用**：

- 经常出现在区域边界
- 也适合作为大区域内部的小中继

#### 模板 5：结构压缩房

**职责**：

- 把原本很长的推进链压缩为未来的稳定捷径
- 把第一次探索成果固化进世界图

**标准组件**：

- 持久 `Trapdoor` 或正式门
- 快速旅行点或地图确认点
- 明确回返方向的 Gate

**作用**：

- 不只是给奖励
- 而是直接改写之后的路径成本

---

### 2.18 最后落回 Ark：Silksong 最值得示巴星借的，不是平台动作，而是“大图减负规律”

如果只看示巴星首个可玩切片，`Silksong` 最值得借的不是横版平台技巧本身，而是下面这 4 件事：

#### 1. 先给每个房间定义“图节点职责”

先问：

- 这是推进房？
- 区域确认房？
- 轻机关切拍房？
- 结构压缩房？

不要先从“这里想摆什么内容”开始。

#### 2. 让入口方向决定房间第一印象

示巴星未来也完全可以让：

- 从上层管道进入
- 从下层破口进入
- 从钟塔门进入

对应不同的镜头、播报和敌人站位。

#### 3. 周期性提供导航确认，而不是连续堆压探索

尤其是首切片里，如果一直让玩家在声学废墟里摸索而没有锚点，就很容易只剩迷失。

你需要周期性投放：

- 区域播报
- 可读远景
- 小地图确认
- 结构性捷径

#### 4. 把捷径当成正式奖励，而不是附带优化

`Silksong` 给我的最大提醒之一是：

在大型地图里，“以后回这里更快了”本身就是重量很高的奖励。

对示巴星来说，这很适合翻译成：

- 共鸣门重开
- 静音桥接通
- 钟塔升降梯恢复
- 废墟内环通路被重新连上

这些都比单纯掉资源更能增强首切片的“世界被我读懂了”的感觉。

### 2.19 Silksong 代表性房间样本拆解（高可信版）

这里我不做“精确几何复原”，而是做一层对当前资料最诚实、但仍然足够能指导生产的样本拆解。

依据来源主要是：

- `WorldNavigation` 的场景命名与出口命名
- `SceneTransitionZoneBase`
- `AreaTitleController`
- `FastTravelMapBase`
- `Trapdoor` / `TripWire`
- `MazeMistZone`

也就是说，下面这些样本更接近：

**“这个 scene 在世界图里大概率承担什么职责”**

而不是：

**“我已经看见了它全部地形细节”**

**节奏作用**：

- 示巴星也需要这种“低压功能房”：

- 明确区分“外部世界路径”与“高权重内部节点”

**标准组件推断**：

- 一个普通进出边
- 一个高权重正式门
- 视觉上比普通侧室更集中
- 可能带标题、条件、仪式反馈

**节奏作用**：

- 它是推进节奏里的“重音节点”
- 告诉玩家这里不是普通岔路，而是章节边界

**对 Ark 的翻译**：

- 很适合变成示巴星的：
  - 大钟室前厅
  - 共鸣圣所外环
  - 主线机关入口厅

#### 样本 7：`MazeMistZone` -> 惩罚式重定向房

**已知证据**：

  - 设置临时 respawn marker
  - `RespawningHero = true`
  - `TimePasses()`

**高可信职责**：

- 不是普通过门
- 而是“犯错 / 误入 / 触发特殊状态后，把你重新定向到某个规则化入口”

**标准组件推断**：

- 特殊触发区
- 非常明确的系统重置逻辑
- 带惩罚或成本的返回

**节奏作用**：

- 它把“环境风险”写成正式结构件
- 让迷宫类或异常区域不只是难走，而是有明确的重置语法

**对 Ark 的翻译**：

- 示巴星未来如果做声雾区、失谐带、静默迷廊，这个样本非常值得借
- 重点不是伤害，而是“规则错了就把你重新抛回某个入口”

### 2.21 如果落回示巴星首切片，Silksong 最值得先借哪 3 个样本

如果只选 3 个，我会优先拿：

#### 1. `Town` -> 主枢纽导航房

#### 2. `Room_Town_Stag_Station` -> 结构压缩房

因为首切片如果没有捷径回返，探索很容易显得“只是在重复跑图”。

因为它最适合翻译成你项目自己的“理解即武器”版本：

- 我不只是拿了一张地图
- 我是正式获得了理解这个区域的解释权

换句话说：

- `Minishoot` 能帮你把房间职责排稳
- `TUNIC` 能帮你把理解门槛做深
- `Silksong` 能帮你把大图导航做顺

这三者叠起来，才是真正适合示巴星首切片的组合。

### 2.22 把 7 类 Silksong 样本继续下沉：每类房间的标准组件清单 + 进出规则 + 奖励规则

下面这一节会尽量保持和前面 `Minishoot` 同样的生产颗粒度。

但需要先强调一个差异：

`Silksong` 这类大地图平台银河城里，很多房间最重要的奖励不是资源，而是：

- 导航权
- 解释权
- 回返压缩权

所以你会看到，这里的“奖励规则”经常不是掉一个东西，而是让整个世界变得更顺手。

---

#### 样本 1：教学推进房（`Tutorial_01` 型）

**标准组件清单**：

- 1 个明确出生点或落点
- 2-3 个可见出口，其中至少 1 个当前可走
- 1 个不会过度压迫的镜头锁区
- 1 组低复杂度平台动作或移动教学
- 0-1 个轻战斗或轻危险
- 1 个能提示“世界不只一条线”的旁向出口

**进出规则**：

- 进入条件：
  - 玩家处于区域初次学习阶段
  - 不依赖复杂前置能力或钥匙
- 退出条件：
  - 玩家理解：
    - 房间节点存在多个出口
    - 方向是有意义的
    - 当前主路是哪一条
  - 不宜在这里放高失败惩罚或强封门

**奖励规则**：

- 奖励以“世界阅读建立”优先
- 最适合给：
  - 第一次方向感
  - 第一处远景承诺
  - 第一条未来分支预告
- 不适合给高价值资源，因为玩家此时还没建立资源价值判断

---

#### 样本 2：主枢纽导航房（`Town` 型）

**标准组件清单**：

- 4 个以上语义明确的出口
- 1 个高辨识度中心安全区
- 1 个区域标题或强命名系统
- 2 个以上功能侧室入口
- 1 个可视上足够稳定的主地标
- 0-1 个地图、快速旅行或主系统入口

**进出规则**：

- 进入条件：
  - 通常位于一个大区域的认知中心
  - 玩家会从多个方向反复回到这里
- 退出条件：
  - 玩家应能从这里做出路线决策
  - 每个出口最好都有不同语义：
    - 主推进
    - 功能侧室
    - 快速旅行
    - 特殊门槛
    - 回返路

**奖励规则**：

- 奖励以“导航减负”优先
- 最适合给：
  - 区域定位
  - 入口记忆
  - 系统汇聚
  - 低压恢复
- 不建议把重心放在 loot 上，否则会削弱它作为结构锚点的价值

---

#### 样本 3：结构压缩房（`Room_Town_Stag_Station` 型）

**标准组件清单**：

- 1 个正式交通或快速旅行交互件
- 1 个当前地点确认组件
- 1 个清晰入口
- 1 个清晰返回主世界的出口
- 0-1 个轻服务功能
- 1 组明显区别于普通通道房的构图

**进出规则**：

- 进入条件：
  - 玩家已经完成一段足够长的推进链
  - 此处应当具有“终于建立永久连接”的心理重量
- 退出条件：
  - 玩家完成：
    - 交通点解锁
    - 节点确认
    - 与主图重新接驳
  - 之后再次经过时，这个房间应当明显更轻、更快

**奖励规则**：

- 奖励以“路径成本下降”优先
- 最适合给：
  - 快速旅行解锁
  - 电梯 / 车站 / 中继点恢复
  - 一段长路径被永久压缩
- 这类房最重要的奖励不是资源，而是“以后来这里更快了”

---

#### 样本 4：制图确认房（`Room_mapper` 型）

**标准组件清单**：

- 1 个单入口或单主入口安全侧室
- 1 个地图 / 制图 / 认知确认交互件
- 1 个足以停下来的观察窗口
- 0-1 个叙事或身份强化物件
- 低敌压或完全无敌压
- 1 个可与外部区域形成认知对照的出口

**进出规则**：

- 进入条件：
  - 通常位于一段探索之后
  - 玩家此时已经产生“我到底在哪”的问题
- 退出条件：
  - 玩家获得某种正式解释权：
    - 地图精度
    - 区域名字
    - 路径读法
  - 不宜立刻把玩家扔进高压战斗，最好先给一点认知消化空间

**奖励规则**：

- 奖励以“认知落地”优先
- 最适合给：
  - 地图更新
  - 区域结构说明
  - 未来目标的重新命名
- 如果给物资，应该是附带的；真正的主奖励是“我终于懂这里了”

---

#### 样本 5：生活侧室房（`Room_shop` / `Room_Mender_House` 型）

**标准组件清单**：

- 1 个与主枢纽短连接的入口
- 1 个非战斗互动主体
- 1 组安全构图或室内构图
- 0-1 个次级门或内侧小间
- 1 个明显区别于外部压力区的氛围层
- 0-1 个轻叙事信息点

**进出规则**：

- 进入条件：
  - 最适合插在导航锚点附近
  - 让玩家在大区域高压之间有短促换气点
- 退出条件：
  - 玩家完成：
    - 补给
    - 交易
    - 修复
    - 聊天 / 得知信息
  - 离开后应自然返回主枢纽或主路，而不是继续嵌套太多复杂结构

**奖励规则**：

- 奖励以“情绪恢复与功能恢复”优先
- 最适合给：
  - 补给
  - 服务
  - 线索
  - 氛围缓冲
- 不适合承载重章节推进，否则会冲淡它作为呼吸房的作用

---

#### 样本 6：章节门槛房（`Room_temple` 型）

**标准组件清单**：

- 1 个高可见度正式门槛
- 1 个普通出入口
- 1 个强调仪式感或章节感的构图焦点
- 0-1 个条件判断或门锁载体
- 0-1 个守门战 / 前哨压力
- 1 个失败后仍然可理解的反馈

**进出规则**：

- 进入条件：
  - 玩家通常先看到门槛，再理解其重量
  - 最好让玩家尽早知道“这里以后要回来”
- 退出条件：
  - 若未满足条件：
    - 给出清晰不可进反馈
    - 保留记忆标签
  - 若满足条件：
    - 通过正式门
    - 进入高权重内部段落

**奖励规则**：

- 奖励以“章节许可”优先
- 最适合给：
  - 主线推进权
  - 新区域进入权
  - 世界规则被承认的反馈
- 不建议只放普通资源，因为那会严重降低门槛房的分量

---

#### 样本 7：惩罚式重定向房（`MazeMistZone` 型）

**标准组件清单**：

- 1 个特殊触发区
- 1 套明确的重置 / 传送逻辑
- 1 个目标重生点或规则化入口
- 1 个能让玩家意识到“这里规则不同”的前置提示
- 0-1 个时间 / 状态 / 记忆成本
- 1 个将惩罚写成世界规则的反馈

**进出规则**：

- 进入条件：
  - 玩家误读规则、走错路径，或进入特殊异常区
  - 不必总是致死，但必须足够明确
- 退出条件：
  - 玩家不会从原地“糊弄过去”
  - 而是被系统性送回某个入口、某个标记点或某种起算状态
  - 回来后应当更容易理解正确读法

**奖励规则**：

- 奖励不以正反馈资源为主，而以“规则教育”优先
- 最适合给：
  - 对区域规则的重新理解
  - 对正确入口的重新记忆
  - 对异常区边界的正式学习
- 这类房的关键不在赚，而在学

### 2.23 一个对 Ark 非常重要的补充：Silksong 的奖励规则，本质上偏“结构性奖励”

如果把这 7 类样本横着看，会发现 `Silksong` 的奖励重心非常少落在“打完给一堆东西”上。

它更常见的高价值奖励是：

#### 1. 导航奖励

- 我知道自己在哪
- 我知道哪些门通向什么
- 我知道这个区域如何命名和记忆

#### 2. 压缩奖励

- 我以后回来更快
- 一条苦路变成短路
- 一个节点被并入我的熟练回返网络

#### 3. 解释奖励

- 我理解这片区域的结构了
- 我有了地图
- 我知道这个门以后该怎么开

#### 4. 许可奖励

- 我被允许进入更深区域
- 世界承认我已经完成前置证明

换句话说，`Silksong` 很适合提醒 Ark：

**在大地图里，最贵的奖励往往不是资源，而是让世界变得更可控。**

下面这张表的用途不是继续分析 `Silksong`，而是把它压成真正能服务示巴星搭房的检查工具。

如果说前面 `Minishoot` 那张表更适合检查：

- 房间职责是否清晰
- 节奏是否稳定
- 奖励是否匹配风险

那么这一张 `Silksong` 版检查表更适合检查：

- 房间有没有承担导航减负
- 房间有没有建立路径解释权
- 房间有没有把探索成果压缩成未来的回返便利

也就是说，它更适合拿来处理：

**“地图一旦开始变大，房间怎样才不会把玩家带进纯迷路疲劳。”**

---

#### A. 通用总检查表（Silksong 版）

每个示巴星房间，无论类型，都先过这一轮：

| 检查项 | 必问问题 | 合格标准 |
| --- | --- | --- |
| 图节点身份 | 这个房间在整张路网里到底是什么节点？ | 必须明确属于引导 / 锚点 / 压缩 / 确认 / 门槛 / 功能 / 重置之一 |
| 入口语义 | 玩家是从哪一边进入，它的第一印象应该是什么？ | 必须说明入口方向与第一阅读版本 |
| 主方向可读性 | 玩家进来后能否立刻判断主路大致方向？ | 至少 1 条主路径清晰，不靠乱试 |
| 导航减负职责 | 这个房间有没有降低迷失成本？ | 至少承担定位 / 命名 / 地标 / 地图 / 捷径之一 |
| 回返价值 | 玩家第二次再来时，这个房间会不会更轻？ | 必须说明它在回返阶段是否变成捷径、确认点或压缩点 |
| 记忆锚点 | 玩家离开后会怎样记住它？ | 至少有 1 个可复述的地标、构图、门槛或机制特征 |
| 奖励类型 | 它发的是哪种奖励？ | 必须明确属于导航 / 压缩 / 解释 / 许可 / 资源 / 情绪恢复之一 |
| 节奏位置 | 它处在切片节奏的哪一拍？ | 明确位于引导 / 推进 / 轻切拍 / 确认 / 压缩 / 门槛中的一个位置 |
| 标准件依赖 | 它依赖哪些可复用标准件？ | 尽量由入口、镜头、功能点、捷径件、重置件组成，而不是专属脚本堆叠 |
| MVP 边界 | 如果先做最小可玩版，最少保留什么？ | 必须能缩成一个不失真的最小版本 |

#### B. 7 类房间专项检查表

| 房间类型 | 必备组件 | 入口必须满足 | 退出必须满足 | 首选奖励 | 生产时最常见错误 |
| --- | --- | --- | --- | --- | --- |
| 初入废墟引导房 | 出生点、主方向出口、轻镜头收束、低复杂度教学点、旁向预告出口 | 玩家第一次进入该区域，无复杂前置条件 | 玩家已理解主方向与未来分支存在 | 导航奖励 / 信息奖励 | 上来就塞战斗、谜题、资源，导致引导失焦 |
| 钟城主锚点房 | 多语义出口、中心安全区、区域命名、主地标、1-2 个功能入口 | 玩家会从多个方向反复回流 | 玩家能明确做路线决策 | 导航奖励 / 情绪恢复奖励 | 所有出口都做成同语义通道，结果不是枢纽只是大厅 |
| 共鸣中继站房 | 正式交通点、当前位置确认、清晰进出边、压缩构图 | 玩家刚完成一段较长探索 | 玩家完成交通解锁并与主图重接 | 压缩奖励 | 只把它当补给点，没有真正缩短回返成本 |
| 测绘确认房 | 单入口安全侧室、测绘交互件、观察窗口、低敌压环境 | 玩家已经产生“我到底在哪”的疑问 | 玩家获得某种正式解释权后离开 | 解释奖励 | 做成纯 lore 房，没有结构读图收益 |
| 低压功能房 | 安全构图、非战斗互动、短连接入口、可选轻叙事点 | 通常邻接导航锚点或主路 | 玩家补给/交易/恢复后自然回主路 | 情绪恢复奖励 / 功能奖励 | 承担过多主线重量，破坏它作为换气点的职责 |
| 大钟门槛房 | 高可见门槛、条件载体、仪式性构图、失败反馈 | 玩家先看到，再知道暂时进不去 | 未满足条件时留下记忆；满足后正式通过 | 许可奖励 / 长期承诺奖励 | 门太轻，既不值得记也不值得回来 |
| 失谐迷廊重置房 | 特殊触发区、重置逻辑、规则化返回点、异常提示 | 玩家误读规则或进入异常区 | 玩家被系统性送回正确入口或起算状态 | 解释奖励 / 规则教育奖励 | 只有惩罚没有学习，玩家只会觉得烦 |

---

#### C. 示巴星首切片专用简化版（Silksong 取向）

如果这张表只服务示巴星首个可玩切片，我建议把填写内容压成下面 8 项。  
每做一个 `SH-SLICE-Rxx`，至少要写满这 8 项：

| 字段 | 要填写的内容 |
| --- | --- |
| 房间 ID | 例如 `SH-SLICE-R05` |
| 房间类型 | 7 类之一 |
| 图节点职责 | 例如“主锚点”“结构压缩”“解释确认” |
| 入口语义 | 玩家从哪边来，这一眼应该读到什么 |
| 必备组件 | 最少 3-5 个，必须是具体组件 |
| 退出条件 | 玩家凭什么能离开这里 |
| 奖励结算 | 玩家获得了什么结构性收益 |
| 回返变化 | 第二次回来时，这个房间和第一次有什么不同 |

---

#### D. 一个建议的填写范例

```text
房间 ID：SH-SLICE-R05
房间类型：共鸣中继站房
图节点职责：把前半段废墟探索压缩成未来稳定捷径
入口语义：玩家从外环噪音区回到钟塔脚下，第一眼应读出“这里终于恢复了连通”
必备组件：共鸣升降梯终端、当前地点标识、返回主路出口、低压安全区、一次解锁反馈
退出条件：玩家激活中继站后，可选择立刻回主路或以后快速回返
奖励结算：结构压缩奖励，之后重跑前半段路线的成本显著下降
回返变化：第一次来是奖励节点，第二次来变成高效率交通节点
```

---

#### E. 最后的执行纪律（Silksong 版）

这张检查表最重要的不是“房间想法很多”，而是强迫每个房间回答下面这 4 个问题：

1. **它在整张地图里到底是哪个节点？**
2. **它第一次进入和第二次回返，语义有没有变化？**
3. **它到底让玩家更懂了什么，或更省了什么？**
4. **它的奖励究竟是资源，还是结构性收益？**

只要这 4 个问题里有两个答不清，这个房间就很可能还只是“概念成立”，还没有进入“可生产”状态。

## 3. TUNIC

### 3.1 先说结论：TUNIC 的强项不是“谜题多”，而是把 **空间门槛、视角门槛、知识门槛** 压成了同一种世界语言

如果用一句话概括 `TUNIC` 的关卡架构，我会写成：

> **Scene Portal 跨场景缝合世界，DungeonRoom 在场景内做可见性切片，Campfire 固定回返锚点，Door / Prayer / Conduit / Manual 把“能不能去”升级成“你有没有学会理解这个世界”。**

它和 Minishoot 的核心区别是：

- `Minishoot` 更像“空间 + 遭遇 + 锁钥”的稳定语法
- `TUNIC` 更像“空间 + 视角误导 + 知识解码”的分层语法

也就是说，`TUNIC` 最值得 Ark 借的不是某一个机关，而是：

**把“知道”本身做成通行证。**

---

### 3.2 证据边界说明

这次 `TUNIC` 的材料不是完整 Unity 场景导出，而是 `il2cpp` 反编译产物，主要依据包括：

- `SceneLoader`
- `ScenePortal`
- `PlayerCharacterSpawn`
- `DungeonRoom`
- `Campfire`
- `Door` / `HolySealDoor` / `TempleDoor`
- `Ladder` / `Elevator`
- `ConduitNode` / `ConduitTeleporter` / `Fuse`
- `PagePickup` / `PageDisplay`
- `GameHelpManager` / `PageCondition` / `SceneCondition`
- `Profile` 中的 `last campfire scene name`、`last page viewed`、`last spawn scene name`

因此下面的结论会偏 **系统架构与门槛语法**，而不是逐房间考古。

---

### 3.3 第一层：世界不是一整张图，而是“跨场景跳转 + 场景内局部切片”两级结构

从 `SceneLoader`、`ScenePortal`、`PlayerCharacterSpawn` 看，`TUNIC` 的世界不是一个大场景硬撑到底，而是：

#### A. 跨场景层：`SceneLoader + ScenePortal + PlayerCharacterSpawn`

- `SceneLoader.LoadScene(string sceneName)` 负责真正切场
- `ScenePortal` 负责：
  - `destinationSceneName`
  - `optionalIDToSpawnAt`
  - `spawnTransform`
  - departure / arrival 过场
- `PlayerCharacterSpawn` 负责：
  - `id`
  - `portalIDToSpawnAt`
  - `GetCurrentSceneName()`
  - `GoTo(string destinationSceneName, string destinationID)`

这意味着 `TUNIC` 的跨区域结构不是模糊跳转，而是：

**“去哪个场景” + “在那个场景的哪个 arrival ID 出生”** 两段式定义。

这很重要，因为它让设计师可以稳定地控制：

- 你从哪条边进入一个区域
- 你在新区域里第一次看见什么
- 返回时镜头朝向与空间解读是否变化

#### B. 场景内层：`DungeonRoom`

`DungeonRoom` 是这次最关键的发现之一。它说明 `TUNIC` 在场景内还做了第二层切片，而不是把一个场景当成完全连续平面。

字段非常能说明问题：

- `farRooms`
- `maskObject`
- `ignoreWhenInConduit`
- `instantRoomTransition`
- `TransitionSequenceRunning`
- `TransitionFade`
- `CurrentRoom`
- `numberOfMyTriggersPlayerIsIn`
- `kDefaultLayer = 0`
- `kHiddenLayer = 15`

这基本可以推断出：

- 一个场景内部会再被拆成多个 `DungeonRoom`
- 玩家进入某个 room trigger 后，当前房间与远房间的可见性、遮罩、灯光、音频会被切换
- 房间之间不是纯几何连续，而是存在“被控制的可见性语法”

也就是说，`TUNIC` 的伪等距秘密感并不只是美术画法，而是**运行时可见性管理**在帮忙。

---

### 3.4 第二层：TUNIC 的“门”不是一种门，而是一整套分层门槛系统

从 `Door`、`HolySealDoor`、`TempleDoor`、`ItemRecepticle`、`ToggleGraveyardGate` 可以看出，`TUNIC` 的锁不是单一 key-door，而是至少有 4 层：

#### A. 常规资源门：`Door`

`Door` 直接绑定：

- `keyItem`
- `stateVariable`
- `useKeyPromptLanguageLine`

这是一层最基础的“有钥匙就开”的资源门。

#### B. 世界状态门：`HolySealDoor`

`HolySealDoor` 绑定：

- `cubes`
- `cubeOpenOrigin`
- `stateVariable`

这不是普通门，而是明显具有仪式感和章节感的世界状态门。  
它的价值不在“挡住玩家”，而在于：

- 告诉玩家这里是大权重边界
- 用形态变化强化“世界被改写了”

#### C. 主线证明门：`TempleDoor`

`TempleDoor` 的字段极有代表性：

- `templeDoorOpen`
- `visitedSpiritRealmForDoorVision`
- `bell_east`
- `bell_west`
- `allHexagonsPlaced`
- `ringBellsMessage`

这说明 Temple Door 不是单条件门，而是**多阶段主线状态汇总门**。  
它把“是否有资格进入”建立在多个世界行为已完成的基础上。

#### D. 组合条件门：`ItemRecepticle` / `ToggleGraveyardGate`

- `ItemRecepticle` 把“交付特定物品 -> 改变状态变量”做成标准件
- `ToggleGraveyardGate` 把 `stateVar` 和 `isNightstateVar` 做了组合判断

这意味着 `TUNIC` 的门槛系统不只是单一布尔值，而是：

**物品、时间/相位、剧情状态、行为证明，可以被组合成同一种空间阻挡语言。**

---

### 3.5 第三层：路径不是纯走路，而是“高程 + 祈祷 + 导线网络”共同构成

#### A. 垂直阅读：`Ladder` / `Elevator`

`Ladder` 的字段已经说明它不是装饰：

- `height`
- `engageRadius`
- `offset`
- `ClimbOn(LadderEnd ladderEnd)`
- `AtBottom()` / `AtTop()`

这意味着梯子本身就是一套精确定义的空间切换标准件。  
它的关卡价值在于：

- 把高低差从“背景层次”变成“路径权限”
- 让玩家在伪等距视角下重新理解前后景关系

`Elevator` 更进一步，它同时受两套条件约束：

- `nodeForPower`
- `IConduitStateListener`
- `IMonsterStateListener`
- `allMonstersDead`
- `powered`

也就是说，电梯不只是移动平台，它还是：

- 动力网络结果
- 战斗清场结果

两者叠加后的空间许可。

#### B. 祈祷-导线网络：`ConduitNode / Fuse / ConduitTeleporter`

这部分是 `TUNIC` 非常独特的一层。  
从类名和字段看，它把“能量网络”和“空间网络”绑定在一起：

- `ConduitNode`
  - `internalConnections`
  - `externalConnection`
  - `accelerator`
  - `CheckConnectedToPower()`
- `Fuse`
  - `isLockingFuse`
  - `optionalSetTrueOnPrayerComplete`
- `ConduitTeleporter`
  - `usableAccordingToConduitNetwork()`
  - `realWorldConduitNode`
  - `kTransitVoidSceneName = "Transit"`

这说明 `TUNIC` 的某些通路、传送、机关，并不是按“拿钥匙开门”的思路实现，而是按：

**祈祷激活 -> 导线通电 -> 网络生效 -> 空间才可用**

去组织的。

这类设计的优势是：

- 玩家感知到自己不是在“开 UI 锁”
- 而是在“恢复世界结构”

---

### 3.6 第四层：Campfire 不是普通存档点，而是世界回返语法中心

从 `Campfire`、`PlayerCharacterSpawn`、`Profile` 看，`TUNIC` 的篝火承担了远比“存档点”更重的结构职责。

`Campfire` 本身负责：

- `spawnTransform`
- `checkpointUnlockedEffect`
- `RespawnAtLastCampfire()`
- `setMostRecentCampfire()`
- `isMostRecentCampfire()`

`Profile` 则明确保存：

- `last campfire scene name`
- `last spawn scene name`
- `last shop arrival scene name`
- `bloodstain scene name`
- `last page viewed`

这意味着篝火至少同时承担：

1. **死亡回返锚点**
2. **空间记忆锚点**
3. **场景切换后的稳定恢复点**
4. **系统层持久化坐标**

所以 `TUNIC` 的安全点不是“让你喘口气”，而是：

**把整个复杂世界重新锚定到一组可记忆、可回返、可存档的节点上。**

这也是为什么它即便复杂，玩家仍然不会彻底失去世界感。

---

### 3.7 第五层：TUNIC 真正的王牌是把“知识”做成了关卡系统，而不是文本系统

这是最值得 Ark 认真研究的一层。

#### A. 页面不是 lore collectable，而是进度对象

`PagePickup` 明确包含：

- `pageName`
- `enablingStateVar`
- `onlyPickupableIfThisInactive`
- `OnPageWasPickedUp`

说明页面拾取是有前置条件、状态反馈和全局回调的系统性行为。

#### B. Manual 不是菜单，而是可操作的阅读空间

`PageDisplay` 不只是个 UI 页签，它有非常完整的“书本观看器”结构：

- `pageRenderers`
- `pageRenderingCamera`
- `pageSpineRoot`
- `pageViewingCamera`
- `pageTurnDuration`
- `zoomedFOV`
- `panSpeed`
- `youAreHereTexture`
- `showSpecificPageOnOpen`
- `specificLeafToShowOnOpen`
- `kPathToPages = "PagePrefabs/leaf_"`

这说明手册在 `TUNIC` 里不是一个附属菜单，而是一个被精心制作的**交互阅读空间**。

#### C. 更关键的是：知识被写进条件系统

`GameHelpManager` 解析的条件头里直接出现：

- `avail_scene_OR`
- `avail_page`
- `done_scene_OR`
- `done_zone_OR`
- `done_cnode`
- `done_page`

并且存在：

- `PageCondition`
- `SceneCondition`
- `ZoneCondition`
- `ConduitNodeCondition`

这等于明说：

**页数、到过哪些场景、进过哪些区域、激活过哪些导线节点，都是同一级别的进度条件。**

再结合：

- `Profile.KnowsTheCross`
- `"[Heir] final boss received interaction, but betrayal=true && manual is incomplete."`

可以非常有把握地说：

`TUNIC` 的终局与高层解法，并不是单纯靠角色数值，而是靠玩家是否真正“学会了阅读这个世界”。

这就是它最强的地方。

---

### 3.8 因此，TUNIC 的关卡并不是“地图 + 谜题”，而是三层门槛叠加

把上面所有层压缩后，可以得到 `TUNIC` 的真实结构：

#### 1. 空间门槛

- 门
- 梯子
- 电梯
- 传送点
- 场景入口/出口

#### 2. 世界状态门槛

- Bell
- Hexagon
- Day/Night
- Fuse / Conduit
- Campfire / Spawn / Spirit World 状态

#### 3. 知识门槛

- Page
- Manual
- 特定认知状态（如 `KnowsTheCross`）
- 玩家对符号、路径、镜头欺骗的再解释能力

这三层门槛的厉害之处在于：

**它们在玩家体感里不是三套系统，而是一件事。**

玩家会觉得自己只是“终于看懂了”，但系统层其实已经完成了：

- 路径理解
- 规则理解
- 进度条件满足

三件事的合一。

---

### 3.9 对 Project Ark 最值得借的 6 条长处

#### A. 把“可见性”当成关卡资产，而不是纯美术问题

`DungeonRoom.maskObject`、`farRooms`、`kHiddenLayer`` 这类字段说明：  
TUNIC 会主动控制玩家此刻该看到什么、不该一眼看到什么。

对 Ark 的启示：

- 示巴星不应只靠 Tilemap 堆细节
- 要主动定义：
  - 哪些信息此刻可见
  - 哪些信息要被钟乳、桥体、塔影、前景结构遮住

#### B. 一个场景内也可以继续切“房间状态”

`TUNIC` 证明：即使不切成很多 `.unity`，场景内部也完全可以用 `DungeonRoom` 级别的状态切换控制节奏。

对 Ark 的启示：

- 你未来的单场景 + 房间系统路线是对的
- 但房间不仅要管敌人和门，还要管：
  - 可见性
  - 灯光
  - 音频
  - 遮罩

#### C. 大门槛必须有仪式感

`TempleDoor`、`HolySealDoor` 这类对象告诉我们：

- 重要门槛不能只是“if true then open”
- 必须有：
  - 前置观看
  - 条件积累
  - 仪式化进入

#### D. 安全点必须兼任世界锚点

`Campfire` 的价值不只是恢复，而是把复杂世界重新锚在一个能回忆、能复活、能再出发的位置。

对 Ark 的启示：

- Checkpoint 不能只是 save point
- 它必须是：
  - 导航点
  - 节奏缓冲点
  - 结构记忆点

#### E. 知识不是文本奖励，而是通行证

这是 `TUNIC` 最大的长处。

对 Ark 的启示：

- 《静默方舟》的“理解即武器”如果想真正落地，就不能只体现在 lore 文本
- 必须让某些区域、门槛、解法、战斗读法真正依赖“玩家是否理解”

#### F. 系统提示也应受知识进度控制

`GameHelpManager` 里把 `scene / zone / page / conduit node` 混成同一种条件体系，意味着提示系统不是外挂，而是世界状态的一部分。

对 Ark 的启示：

- 未来示巴星的提示、NPC 台词、地图标注、广播内容，可以根据：
  - 到过哪个区
  - 听过哪段录音
  - 获得哪类共鸣知识
  - 触发过哪个世界节点

来动态变化

---

### 3.10 对 Ark 的直接映射：TUNIC 最适合借的不是“形式”，而是 4 套生产规则

#### 规则 1：每个重要区域至少有一层“看不全”

不要把所有路径信息平铺给玩家。  
用前景遮挡、纵向落差、桥体、钟楼投影、音波雾幕制造“部分可见”。

#### 规则 2：每个大门槛都要同时满足“看见过 + 理解过 + 将来可回返”

这才会形成真正的银河城承诺，而不是一次性锁门。

#### 规则 3：Checkpoints 必须承担空间记忆职责

示巴星的每个主要安全点都应该让玩家能回答：

- 我现在在哪个章节
- 我能从这里去哪里
- 如果死了我会回到哪里

#### 规则 4：至少有一类门槛必须不是“钥匙门”，而是“理解门”

这类门可以是：

- 听懂某段钟律后才能判断的隐藏路
- 读出某种材料变化规律后才敢通过的黑水区
- 理解“静默不是安全，反而是陷阱”的反直觉关卡

只有这样，Ark 才会真正长出自己的 `TUNIC` 式辨识度。

---

### 3.11 一个总总结：TUNIC 不是在设计房间，它是在设计“认知重读”

TUNIC 最厉害的地方不是地图复杂，也不是谜题精巧，而是：

**同一个世界会因为你获得了新的理解，而被重新阅读。**

这和传统银河城“拿到二段跳所以能上去”不同。  
它更进一步地把：

- 新能力
- 新视角
- 新知识
- 新解释

全部做成了关卡再解锁的一部分。

如果 Minishoot 教 Ark 的是：

> “如何把房间组织得稳定、清晰、可生产”

那么 TUNIC 教 Ark 的就是：

> “如何让同一个房间在玩家脑中前后变成两张地图。”

### 3.12 TUNIC 机制词典：它到底用哪些标准件在制造“认知重读”

下面这部分的目标和前面 Minishoot 的机制词典一样：

- 不再只讲理念
- 而是整理 `TUNIC` 真实存在的“空间标准件”

虽然这次没有完整场景 YAML，但从类结构看，下面这些部件已经足够还原它的核心关卡语法。

#### A. 跨场景入口件：`ScenePortal` / `ShopScenePortal` / `PlayerCharacterSpawn`

这组对象负责世界图上的“大折页”。

**可确认职责**：

- 把世界拆成多个 scene
- 通过 `destinationSceneName + optionalIDToSpawnAt` 精确控制进入点
- 通过 `PlayerCharacterSpawn.id` 与 `portalIDToSpawnAt` 精确控制落点
- 在 `ShopScenePortal` 中，甚至还会因为 arrival scene 不同而翻转镜头/到达朝向

**设计意义**：

- 同一个出口，从不同方向回来，玩家看到的不是同一幅图
- 入口本身就是叙事控制器和空间控制器

**对 Ark 的映射**：

- 未来 `RoomTransition` 不应只知道“去哪个房间”
- 还应知道“以什么姿态、什么方向、什么镜头状态进入那个房间”

---

#### B. 场景内切片件：`DungeonRoom` / `DungeonRoomRigidbodyTracker`

这是 `TUNIC` 最接近“房间系统”的核心标准件。

**可确认职责**：

- `DungeonRoom` 用 trigger 判断玩家进入/离开
- 维护 `CurrentRoom`
- 通过 `maskObject`、`farRooms`、`kHiddenLayer` 控制当前房与远房间的显隐
- `DungeonRoomRigidbodyTracker` 说明：进入当前 room 的不只是静态场景，连动态物件也会跟着房间显隐语法走

**设计意义**：

- 玩家并不是总在看完整空间
- 而是在看“当前允许被阅读的局部真相”

**对 Ark 的映射**：

- 你的 `Room` 不该只负责门和 Encounter
- 它还应该能控制：
  - 前景遮挡
  - 可见远景
  - 局部灯光
  - 动态物件的显隐

---

#### C. 回返锚点件：`Campfire` / `ReturnToCampfireTrigger` / `Profile`

这组对象共同定义了 `TUNIC` 的“失败后世界如何重新粘合”。

**可确认职责**：

- `Campfire` 负责互动、最近篝火登记、复活点更新、解锁视觉反馈
- `ReturnToCampfireTrigger` 说明某些区域/事件会强制把玩家送回最近锚点
- `Profile` 保存：
  - `last campfire scene name`
  - `last spawn scene name`
  - `last shop arrival scene name`
  - `bloodstain scene name`

**设计意义**：

- 世界很复杂，但失败后的重组方式非常稳定
- 玩家不是被随便扔回某个点，而是被送回“最近一次被世界承认的落脚处”

**对 Ark 的映射**：

- `CheckpointSystem` 未来最好区分：
  - 最近存档点
  - 最近合法出生点
  - 最近商店/安全区入口
  - 最近掉落遗留点

---

#### D. 普通门与权重门：`Door` / `ProximityDoor` / `HolySealDoor` / `TempleDoor` / `ObsidianDoorway`

这部分可以看出 `TUNIC` 不同权重的门槛是如何分层的。

**1. `Door`**

- 交互型资源门
- 绑定钥匙与状态变量
- 是最基础的“消耗资源换通行”

**2. `ProximityDoor`**

- 触发型自动门
- 不要求停下交互
- 更像节奏门，而不是资格门

**3. `HolySealDoor`**

- 立方体分解式开门
- 强视觉仪式感
- 明显属于章节级边界

**4. `TempleDoor`**

- 明显汇总多重世界条件
- 与 Bell、Spirit Realm、Hexagon 等主线状态绑定
- 属于“主线证明门”

**5. `ObsidianDoorway`**

- 纯视觉高权重门框件
- 说明 `TUNIC` 甚至会为“门槛感”单独造视觉类对象，而不是全靠逻辑门

**设计意义**：

- 不是所有门都该长得一样、用法一样、权重一样

**对 Ark 的映射**：

- 以后示巴星的“门”至少应分成：
  - 普通锁门
  - 自动节奏门
  - 章节门
  - 仪式门
  - 视觉承诺门

---

#### E. 祈祷与世界改写件：`PrayerListenerBase` / `ConduitPlatform` / `ConduitTeleporter` / `Fuse` / `PrayerListenerSpiritWarp` / `TimeDevice`

这组对象非常接近 `TUNIC` 自己的“世界法则机关库”。

**可确认职责**：

- `PrayerListenerBase` 是一类统一入口，说明“祈祷”是标准交互范式，不是单例剧情行为
- `ConduitPlatform` 用祈祷激活平台/节点
- `ConduitTeleporter` 受导线网络是否可用约束，还能导向 `Transit` / `RelicVoid` / `Spirit Arena`
- `Fuse` 说明导线网络里存在关闭、锁闭、充能、状态广播
- `PrayerListenerSpiritWarp` 直接把祈祷和 `Spirit World` 连接
- `TimeDevice` 说明“时间/世界相位变化”也是一个标准化可触发行为

**设计意义**：

- `TUNIC` 不把机关做成一堆各写各的脚本
- 它更像先定义一套“世界交互动词”，再让不同对象实现这些动词

**对 Ark 的映射**：

- 示巴星后续如果要做“鸣钟 / 共振 / 静默切换 / 相位切换”
- 最好也先有统一交互动词，而不是每个机关单独写一套逻辑

---

#### F. 隐藏与重读件：`SecretPassagePanel` / `RotatingCubeClue`

这两个小部件很能说明 `TUNIC` 的隐藏方法不是纯随机藏东西。

**可确认职责**：

- `SecretPassagePanel`：可受伤、可破坏、受状态变量持久化
- `RotatingCubeClue`：存在可配置 `sequence`

**设计意义**：

- 隐藏路可以是“物理破坏后永久成立”
- 线索物可以是“反复播放的世界提示”

**对 Ark 的映射**：

- 隐藏路不该只靠墙后藏宝箱
- 还可以靠：
  - 可破裂的钟膜
  - 周期性回声提示
  - 某段可观察的材料变化规律

---

#### G. 知识门槛件：`PagePickup` / `PageDisplay` / `GameHelpManager` / `GHEntry` / `PageCondition`

这是 `TUNIC` 最独特的一组“非传统关卡件”。

**可确认职责**：

- `PagePickup` 是持久化进度对象
- `PageDisplay` 是完整阅读器而不是普通 UI
- `GameHelpManager` 会按：
  - `avail_scene_OR`
  - `avail_page`
  - `done_scene_OR`
  - `done_zone_OR`
  - `done_cnode`
  - `done_page`
  组织条件
- `GHEntry` 将活动、任务、子任务都绑定在 condition 列表上

**设计意义**：

- 场景进度、区域到达、导线激活、页面获取，全都被视为同一套“理解进度”

**对 Ark 的映射**：

- 未来 `WorldKnowledge` 或 `LoreProgress` 最好不要只记录“读过文本没有”
- 应该能和：
  - 去过的区
  - 触发过的共鸣点
  - 理解过的机制类别
  - 开启过的世界节点
  进入同一套条件系统

---

### 3.13 TUNIC 的房间组织法则：它更像“视角句法”，不是“战斗句法”

如果说 Minishoot 的房间组织更像：

> 过路房 -> 压力房 -> 清算房 -> 回报房 -> 安全点

那么 `TUNIC` 更像：

> 看见房 -> 怀疑房 -> 验证房 -> 重读房 -> 回返房

这里的重点不是敌人，而是认知状态。

#### 法则 1：先让玩家“看见”，但不要让玩家立刻“看懂”

这是 `TUNIC` 最核心的空间策略。

典型支撑件：

- `ScenePortal`
- `DungeonRoom.maskObject`
- `farRooms`
- 高程切换件（`Ladder` / `Elevator`）

**效果**：

- 你会先看到一个地方
- 但你未必知道怎么过去、从哪过去、现在能不能过去

#### 法则 2：高权重节点一定会被重复读取

篝火、Temple Door、Holy Seal Door、Conduit Teleporter 这类东西，不是一次性看的。

它们会经历至少两个阶段：

1. 第一次看见，但不能使用
2. 未来带着新条件回来，重新解释

这就是 `TUNIC` 的“认知回路”。

#### 法则 3：安全点后面通常不是结尾，而是新一轮阅读的起点

`Campfire` 的意义不是切断流程，而是重置玩家的空间理解成本。

它会让玩家从一个新的稳定点重新扩散。

#### 法则 4：某些房间的奖励不是物资，而是“解释权”

拿到 Page、理解符号、知道某条隐藏规则，本质上是在获得：

- 新路径解释权
- 新符号解释权
- 新门槛解释权

这是 `TUNIC` 和普通资源驱动关卡最大的不同。

#### 法则 5：世界状态变化会反向改变旧区域的阅读方式

从 `Spirit World`、`TimeDevice`、`ToggleGraveyardGate`、`ConduitTeleporter` 可以看出：

- 旧地形不是静态旧地形
- 它会因为世界相位、导线状态、认知变化而变成新内容

所以 `TUNIC` 的 replay value 很大一部分不是“新房间”，而是“旧房间重读”。

---

### 3.14 把 TUNIC 压成 8 类可生产房间模板

虽然我们现在没有完整场景逐房间数据，但从系统层已经足够抽出 8 类高可信模板。

#### 模板 1：远景承诺房

**职责**：

- 让玩家先看见重要目标
- 但不立刻给出完整解法

**常见部件**：

- 高权重门槛对象
- 远景构图
- 遮挡或高低差

#### 模板 2：局部遮蔽房

**职责**：

- 控制玩家此刻能读到的空间信息量

**常见部件**：

- `DungeonRoom`
- `maskObject`
- `farRooms`

#### 模板 3：垂直重读房

**职责**：

- 让玩家通过梯子、电梯、高差重新解释前后景关系

**常见部件**：

- `Ladder`
- `Elevator`
- 多层可视结构

#### 模板 4：祈祷改写房

**职责**：

- 让玩家通过祈祷改变世界状态，而不是捡钥匙开门

**常见部件**：

- `PrayerListenerBase`
- `Fuse`
- `ConduitPlatform`
- `TimeDevice`

#### 模板 5：传导枢纽房

**职责**：

- 把多个导线状态、平台、传送点串成一个高权重网络节点

**常见部件**：

- `ConduitNode`
- `ConduitTeleporter`
- `IConduitStateListener`

#### 模板 6：篝火回返房

**职责**：

- 重置风险
- 固定记忆锚点
- 为下一轮扩散做准备

**常见部件**：

- `Campfire`
- 回返路径
- 明确章节标识

#### 模板 7：知识提示房

**职责**：

- 给玩家解释新规则的片段
- 但通常不一次讲明

**常见部件**：

- `PagePickup`
- `RotatingCubeClue`
- 强符号环境

#### 模板 8：知识解锁房

**职责**：

- 验证玩家是否真正理解某条规则

**常见部件**：

- Page / 状态条件
- 世界门槛
- 隐藏路径或特殊交互

### 3.15 对 Ark 最有价值的一点补充：TUNIC 的“颗粒度”其实不是房间颗粒度，而是“再解释颗粒度”

Minishoot 的颗粒度更适合直接回答：

- 这个房间是过路房还是清算房？
- 这里该不该封门？
- 奖励该怎么发？

而 `TUNIC` 更适合回答：

- 这个房间第一次看见时，玩家应该误解什么？
- 第二次回来时，玩家应该重新理解什么？
- 哪个对象负责承诺，哪个对象负责验证？
- 哪个信息现在该看见，哪个信息现在不该看见？

所以如果你要把 `TUNIC` 真正借到《静默方舟》里，最重要的不是照抄它的门或谜题，而是：

**给每个高权重房间定义“第一次阅读版本”和“第二次阅读版本”。**

这才是它和 Minishoot 在颗粒度上的真正差别。

---

### 3.16 给示巴星的直接翻译问题

为了让 `TUNIC` 的拆解真正服务示巴星，后面你在搭房间时，我建议每个高权重房间都先回答下面 4 个问题：

1. 玩家**第一次看到**这个房间时，会误解什么？
2. 玩家在什么条件下会获得**重新解释它**的资格？
3. 房间里哪个对象承担“承诺”，哪个对象承担“验证”？
4. 这次重读带来的奖励是：
   - 新路径
   - 新安全点
   - 新世界理解
   - 新战斗读法

如果这 4 个问题答不清，那你借到的就只是 `TUNIC` 的表面气质，不是它的结构力量。

---

### 3.17 TUNIC 关卡系统扫尾补遗：把主骨架之外的“缝合件”补齐

这一节的目的，是把前面还没明确展开、但实际上很影响关卡体感的系统补齐。  
它们不一定像 `DungeonRoom` 或 `Campfire` 那样是主骨架，但正是这些部件让 `TUNIC` 的世界读起来不像一堆松散房间。

#### A. 宝箱并不是纯奖励容器，而是“状态化回报节点”

从 `Chest` 可以确认：

- 有 `chestID`
- 有 `openedStateVar`
- 有 `killAllTheseFirst`
- 有 `itemContents` / `itemContentsQuantity`
- 有 `moneySprayQuantity`
- 还有 `OnOpen` 与全局 `OnOpenChest`

这说明 `TUNIC` 的宝箱不是单纯掉落点，而是可以承担多种职责：

- 普通资源回报
- 战后解锁回报
- 条件成立后可领取回报
- Fairy / 特殊表现型回报

也就是说，宝箱在 `TUNIC` 里不是“摆一个箱子”，而是：

**一个带持久化、条件、演出和回报广播的标准奖励节点。**

#### B. 血痕回收不是 UI 惩罚，而是“回返型风险房”

`BloodstainChest` 与 `BloodstainManager` 很说明问题：

- `BloodstainChest` 本身可交互
- 有独立图形根节点和爆炸效果根节点
- `collectBloodstain()`
- `doPushbackBlast()`
- 管理器会在 `onLevelWasLoaded` 时生成/处理

这意味着死亡遗失物的回收不是菜单行为，而是被重新放回世界里，变成一个**必须走回去处理的空间事件**。

对关卡设计的影响是：

- 失败并不会把风险清空
- 风险会重新附着在一个具体地点
- 玩家下一次经过时，对同一路径的读法会改变

这其实也是一种“认知重读”。

#### C. 镜头覆盖不是纯演出，而是空间解释器

`CameraPositionOverrideTrigger` 的字段非常值得重视：

- `overrideTransform`
- `mappingMode`
- `forceLockWhenActive`
- `playerInTriggerCount`
- `BestTrigger(Vector3 position)`
- `MapPosition(Vector3 position)`

这说明镜头并不是被动跟随玩家，而是在特定区域里被主动重映射。  
也就是说，`TUNIC` 的某些空间之所以“看起来像秘密”，未必只是因为地形摆得好，而是因为镜头本身在重新解释这个地形。

对 Ark 的启示非常直接：

- 如果未来示巴星只靠场景摆放，不靠镜头控制，你会很难做出真正稳定的“认知误导”
- 某些高权重房间应该有局部镜头语法，而不是全地图一个跟随摄像机到底

#### D. 灵体与现实之间的出生点不是同一套语义

`PlayerCharacterSpiritSpawn` 说明：

- 它是 `PlayerCharacterSpawn` 的特化版本
- 有 `destinationScene`
- 有 `GetSpiritWorldSpawnID(string realWorldSceneName, string id)`
- 还带 `isUnlocked`

这意味着 Spirit World 不是普通换皮场景。  
它和现实世界之间存在**命名映射过的对应出生点网络**。

这很重要，因为它证明 `TUNIC` 的双世界结构不是“切个场景”，而是：

**同一张世界拓扑，在另一套规则下被重新映射。**

#### E. 床与睡眠提示说明“昼夜切换”被做成了空间行为

`BedToggle` 和 `SleepPrompt` 共同说明：

- 床本身是可互动的出生/切换节点
- 绑定 `canBeUsed`
- 绑定 `isNightStateVar`
- `SleepPrompt` 还记录目标 `scene` / `id` / `visited`

这意味着：

- 昼夜切换不是简单改光照
- 它会影响出生点、可互动性、场景目标和访问语义

这让“时间相位”真正成为关卡层面的东西，而不是氛围层面的东西。

#### F. 区域标题不是 UI 装饰，而是章节锚点

`AreaData` 与 `AreaLabel` 说明：

- 区域有明确的双行标题数据
- `PlayerCharacterSpawn` 还带 `labelToPlayOnArrival`

这意味着章节名出现的时机是被入口系统控制的。  
它不是“走到这差不多弹个字”，而是关卡节奏的一部分。

这对 `TUNIC` 特别重要，因为它在一个复杂、可重读的世界里，不断帮玩家确认：

- 你现在在哪
- 这里是一个什么性质的段落

#### G. 少数特殊隐藏脚本说明：TUNIC 也有区域专属秘密件

像 `TropicalSecret` 这种命名非常直接的对象，说明 `TUNIC` 除了通用标准件之外，确实也有部分区域特供的秘密逻辑。

这件事的启示不是“它到处写特例”，而是：

- 核心 80% 用标准件
- 最后 20% 用少量特殊件给区域建立独特记忆

这和 Minishoot 的方法是相通的，只是 `TUNIC` 把特例更多用在“秘密”和“认知重读”上。

### 3.18 现在可以更准确地描述 TUNIC 的完整关卡标准件层级

如果把前面所有内容合并，`TUNIC` 的关卡标准件可以按 6 层来理解：

#### 1. 世界缝合层

- `SceneLoader`

#### 4. 世界改写层

#### 6. 知识与提示层

- `GameHelpManager`

- `SceneCondition`

负责：

- 知识投放
- 提示可见性
- 条件验证
- 章节命名与认知确认

### 3.19 因此，TUNIC 的“房间样本拆解”不能完全照抄 Minishoot 的拆法

这里要非常明确一点：

如果后面我们继续做 `TUNIC 代表性房间样本拆解`，它不会和 `Minishoot` 一样主要拆：

- Encounter 比例
- 封门节点
- 房间回报节拍

因为 `TUNIC` 的重心不在那里。

它更应该拆的是：

- 这个房间第一次看见时，哪些信息被遮住
- 玩家为何会误读它
- 这个误读未来被什么系统纠正
- 房间是如何通过入口、镜头、世界状态、知识条件重新被解释的

换句话说：

`Minishoot` 的样本拆解更像“空间职责样本”，  
而 `TUNIC` 的样本拆解更像“认知阶段样本”。

---

### 3.20 对 Ark 的下一条高价值问题

现在 `TUNIC` 已经补到这一步，下一步最值得继续的不是再泛读类名，而是开始做：

**“TUNIC 代表性认知样本拆解”**

也就是选 4-6 类最典型的重读结构，比如：

- 远景承诺 -> 未来理解
- 祈祷改写 -> 世界重排
- 篝火回返 -> 风险重挂载
- 页面提示 -> 隐藏路解锁
- 双世界对应出生点 -> 旧区域再解释

这样就能真正做到和 `Minishoot` 一样高的“可用颗粒度”，只是分析维度会不同。

### 3.21 TUNIC 代表性认知样本拆解

> 说明：以下样本不是基于完整场景坐标复原，而是基于已确认的系统结构、状态变量、入口系统、镜头触发与知识条件做出的高可信“认知样本”重建。  
> 目标不是截图考古，而是提炼出 **TUNIC 如何让同一个区域前后读成两张地图**。

---

#### 样本 1：远景承诺 -> 后续证明房

**代表系统**：

- `TempleDoor`
- `HolySealDoor`
- `ObsidianDoorway`
- `ScenePortal`
- `CameraPositionOverrideTrigger`

**第一次阅读版本**：

- 玩家先看到高权重入口或大门槛
- 视觉上已经知道“那里很重要”
- 但当前并不理解：
  - 条件是什么
  - 路径在哪
  - 现在能不能进

**误读点**：

- 容易把它当成“暂时打不开的普通锁”
- 但实际上这通常是主线证明门、章节门、仪式门

**第二次阅读版本**：

- 当玩家拥有新的世界状态或知识后回来
- 这个房间不再只是“看到一个门”
- 而是“终于知道为什么之前进不去”

**承诺对象**：

- 大门
- 门框
- 远景构图
- 特殊镜头

**验证对象**：

- `TempleDoor` 的多状态条件
- `HolySealDoor` 的状态变量

**给 Ark 的翻译**：

- 示巴星也应该有至少 1-2 个这种房
- 重点不是“门锁住了”
- 而是“玩家先被许诺，再被迫成长到有资格”

---

#### 样本 2：局部遮蔽 -> 空间重读房

**代表系统**：

- `DungeonRoom`
- `DungeonRoomRigidbodyTracker`
- `maskObject`
- `farRooms`
- `CameraPositionOverrideTrigger`

**第一次阅读版本**：

- 玩家进入房间时只能读到局部空间
- 远处结构要么被遮住，要么被弱化，要么被镜头改写

**误读点**：

- 玩家会以为“这里只有这么大”
- 或“这条路只通向这一边”

**第二次阅读版本**：

- 当玩家从另一方向进入、在别的高差观看、或经历状态变化后
- 同一空间会暴露出之前看不见的连接关系

**承诺对象**：

- 半可见远景
- 露一角的桥/梯/门
- 被遮住的结构边缘

**验证对象**：

- 新入口方向
- 新镜头位置
- 新高差
- 新的当前房间切片关系

**给 Ark 的翻译**：

- 示巴星的关键房间不应该第一次就交代全图
- 前景钟膜、桥体阴影、音波雾幕都可以承担这个职责

---

#### 样本 3：垂直差 -> 前后景反转房

**代表系统**：

- `Ladder`
- `Elevator`
- `PlayerCharacterSpawn`
- `CameraPositionOverrideTrigger`

**第一次阅读版本**：

- 玩家在平面上看到某些前景/后景结构
- 但无法正确判断它们之间是否相通

**误读点**：

- 伪等距会天然制造“这只是背景”或“这离我很远”的错觉

**第二次阅读版本**：

- 通过梯子、电梯或新出生方向抵达另一个高程
- 玩家才意识到：
  - 那不是背景，是路径
  - 那不是装饰，是未来会走到的位置

**承诺对象**：

- 看得见但站不上去的平台
- 远处楼梯、塔顶、桥面

**验证对象**：

- `LadderEnd`
- `ElevatorStop`
- 另一侧的 spawn / portal

**给 Ark 的翻译**：

- 示巴星的高低差不要只服务美术层次
- 要服务“第一次误判空间，第二次纠正空间”的体验

---

#### 样本 4：祈祷改写 -> 世界重排房

**代表系统**：

- `PrayerListenerBase`
- `ConduitPlatform`
- `Fuse`
- `ConduitNode`
- `ConduitTeleporter`
- `TimeDevice`
- `PrayerListenerSpiritWarp`

**第一次阅读版本**：

- 玩家看到某个平台、传送点、导线、门或机关
- 但它们现在不工作，或者只是场景的一部分

**误读点**：

- 容易把这些对象当作“当前还没拿到钥匙所以不能用”
- 实际上它们很多不是钥匙门，而是世界法则门

**第二次阅读版本**：

- 当玩家学会祈祷、理解导线网络、改变世界相位后
- 同一批对象会突然从“背景装置”变成“可用路径”

**承诺对象**：

- 发光节点
- 断开的导线
- 不响应的传送台
- 静止的平台

**验证对象**：

- Prayer 行为本身
- Conduit 通电状态
- Fuse 完成状态
- Time / Spirit 状态切换

**给 Ark 的翻译**：

- 这几乎是示巴星最值得借的一类
- 未来可做成：
  - 鸣钟改写材料相位
  - 静默切换热传导性质
  - 共振激活旧世界通路

---

#### 样本 5：篝火回返 -> 风险重挂载房

**代表系统**：

- `Campfire`
- `ReturnToCampfireTrigger`
- `BloodstainChest`
- `BloodstainManager`
- `Profile`

**第一次阅读版本**：

- 玩家第一次通过某段危险路时，主要在想“我要不要继续”

**误读点**：

- 会以为死亡只是时间成本
- 但 `TUNIC` 会把损失重新挂回世界，迫使你重走这条路

**第二次阅读版本**：

- 因为血痕/损失回收点的存在
- 同一路径再次经过时，玩家对风险、捷径、敌位、安全点价值的理解都会变化

**承诺对象**：

- 篝火
- 安全点
- 尚未拿回的损失

**验证对象**：

- `RespawnAtLastCampfire()`
- `BloodstainChest.collectBloodstain()`
- 最近 campfire / spawn 持久化

**给 Ark 的翻译**：

- 如果未来《静默方舟》要做“死亡后回收”机制
- 就不该只是地上掉一包资源
- 而应该让回收行为重新塑造玩家对路径的阅读

---

#### 样本 6：页面提示 -> 知识解锁房

**代表系统**：

- `PagePickup`
- `PageDisplay`
- `GameHelpManager`
- `GHEntry`
- `PageCondition`
- `Profile.KnowsTheCross`
- `FinalPagePickup`

**第一次阅读版本**：

- 玩家拿到一页、看到一条提示、读到一段图文
- 但未必立刻知道它能解开什么

**误读点**：

- 很容易把它误认为是 lore collectable
- 但实际上它往往是“世界规则说明书的一角”

**第二次阅读版本**：

- 当玩家把页面知识与现实场景对上号
- 原本不可读的符号、隐藏路、特殊互动 suddenly become legible

**承诺对象**：

- 页面
- 手册图示
- 章节提示
- 区域符号

**验证对象**：

- `PageCondition`
- `done_page`
- `avail_page`
- 特定世界交互是否被玩家正确执行

**给 Ark 的翻译**：

- 这正对应你的“理解即武器”
- 真正重要的是：
  - 知识先作为疑问投放
  - 再作为路径或战斗解法被验证

---

#### 样本 7：双世界对应 -> 旧区域翻面房

**代表系统**：

- `PlayerCharacterSpiritSpawn`
- `PrayerListenerSpiritWarp`
- `visitedSpiritRealmForDoorVision`
- `GetSpiritWorldSpawnID(...)`

**第一次阅读版本**：

- 玩家在现实世界中看到一个区域，只能按当前规则理解它

**误读点**：

- 会以为这个区域的含义已经固定

**第二次阅读版本**：

- 当进入 Spirit World 或通过相位变化重新抵达对应点
- 玩家才意识到：同一世界拓扑，在另一套规则下会变成另一张图

**承诺对象**：

- 异样门槛
- 灵体入口
- 现实世界中解释不通的结构

**验证对象**：

- Spirit spawn 映射
- Spirit warp
- 某些门槛在灵体视角下的重新说明

**给 Ark 的翻译**：

- 如果示巴星未来也做“静默相位 / 共振相位”双重读法
- 那最接近 TUNIC 的借法不是做平行世界剧情
- 而是让**同一空间在不同理解层下长出不同路径语义**

---

### 3.22 把 7 个认知样本压成 4 个最适合 Project Ark 借的模板

如果不想一次记住 7 类，我建议先压成 4 个模板：

#### 模板 A：承诺-验证房

来源样本：

- 样本 1
- 样本 6

用途：

- 先给承诺，再给理解，再给通过

#### 模板 B：遮蔽-重读房

来源样本：

- 样本 2
- 样本 3

用途：

- 让同一空间在不同进入角度/高程下被重新解释

#### 模板 C：改写-通路房

来源样本：

- 样本 4
- 样本 7

用途：

- 通过世界法则或相位变化，让背景设施变成可用路径

#### 模板 D：风险-回返房

来源样本：

- 样本 5

用途：

- 用安全点、死亡损失、回收路径重新塑造玩家的路线判断

### 3.23 最后落回 Ark：示巴星最该借哪三类 TUNIC 样本

如果只考虑示巴星首个可玩切片，我认为最值得优先借的是：

#### 1. 样本 4：祈祷改写 -> 世界重排房

因为它最接近你项目自己的“世界法则型关卡”潜力。

#### 2. 样本 6：页面提示 -> 知识解锁房

因为它最接近“理解即武器”的真正落地方式。

#### 3. 样本 2：局部遮蔽 -> 空间重读房

因为它最便宜，也最容易在首个切片里直接做出来。

换句话说：

- `Minishoot` 更适合帮你解决“房间怎么稳”
- `TUNIC` 更适合帮你解决“房间怎么在玩家脑中变深”

这两者叠起来，才是最接近你现在真正需要的参考组合。

## 4. 跨游戏综合结论

如果把前三部分压成一句真正可执行的话，我会这样总结：

> **`Minishoot` 负责把房间职责与节奏排稳，`Silksong` 负责把大图导航与回返压顺，`TUNIC` 负责把理解门槛与空间重读做深。示巴星首个可玩切片最理想的做法，不是三者各抄一点，而是把它们压成“稳结构 + 顺导航 + 深认知”的同一套房间生产语言。**

---

### 4.1 先给一个总判断：三款游戏分别在补 Ark 的三块短板

如果只用最短的话概括：

- `Minishoot` 补的是：**房间语法**
- `Silksong` 补的是：**路网语法**
- `TUNIC` 补的是：**理解语法**

而 `Project Ark`，尤其是示巴星首切片，当前最需要的正是这三层同时成立：

1. 房间本身要稳，不然切片节奏会散
2. 房间之间要顺，不然玩家会陷入迷路疲劳
3. 房间读完之后要更深，不然“理解即武器”会只剩设定文本

换句话说，这三款参考游戏并不是在回答同一个问题：

- `Minishoot` 回答的是：**“一个房间在流程里该干什么？”**
- `Silksong` 回答的是：**“很多房间连成大图后，怎样仍然可导航、可回返？”**
- `TUNIC` 回答的是：**“怎样让同一个空间在玩家脑中持续变深？”**

这也是为什么三者应该叠加使用，而不是互相替代。

---

### 4.2 哪些长处适合 Ark 直接采纳

这里的“直接采纳”指的是：不依赖原游戏特定品类前提，转译成本低，放到 `Project Ark` 里仍然成立。

#### A. 从 `Minishoot` 直接采纳：房间职责模板化

最值得直接采纳的是：

- 房间类型先行
- 进出条件先行
- 奖励类型先行
- 再去填敌人、互动和叙事

也就是把每个房间都先压成：

- 主职责
- 最低标准组件
- 进入条件
- 退出条件
- 奖励结算

这几乎可以原样成为 Ark 的生产纪律。

#### B. 从 `Silksong` 直接采纳：节点化路网与结构性奖励

最值得直接采纳的是：

- 显式路网思维
- 入口方向决定第一阅读
- 区域锚点周期性减负
- 捷径 / 中继站 / 快速回返本身就是高价值奖励

这些东西和横版/顶视角关系不大，都是大地图成立的基础。

#### C. 从 `TUNIC` 直接采纳：知识先投放，再验证

最值得直接采纳的是：

- 先给玩家疑问
- 再给解释线索
- 最后让解释变成通过权

尤其适合 Ark 的“理解即武器”主轴。

但要注意，直接采纳的是这个结构，不是照搬它的手册谜题形式。

---

### 4.3 哪些长处只能“借思想，不借形式”

这部分很重要，因为三个参考游戏里都有一些很强但不适合直接照搬的部分。

#### A. `Minishoot`：借它的房间秩序，不借它的轻量感外观

不建议直接照搬的是：

- 它那种偏轻快、偏清爽的世界压力密度
- 过于“玩具化”的开放战斗表达

Ark 的气质更重、更压抑，所以更该借它的：

- 节奏秩序
- 房间分工

而不是借它的表面氛围。

#### B. `Silksong`：借它的导航纪律，不借它的平台动作前提

不建议直接照搬的是：

- 依赖高机动平台动作的路径筛选
- 横版视角下的高低差门槛表现方式

Ark 该借的是：

- 路网显式化
- 节点分层
- 回返压缩
- 入口语义

而不是试图把平台动作难点直接翻译成顶视角绕路。

#### C. `TUNIC`：借它的认知门槛，不借它的高信息遮蔽密度

不建议首切片直接照搬的是：

- 过于高密度的误导与隐藏
- 需要大规模手册/符号系统支撑的知识门槛
- 双世界级别的重解释复杂度

首切片更适合借它最轻、最稳的三个思想：

- 远景承诺
- 局部遮蔽重读
- 世界法则改写通路

也就是说：

**先做“玩家能学会”的知识门槛，再考虑“玩家要花很久破解”的知识门槛。**

### 4.4 三款游戏压到一起后，Ark 最该采用的统一生产语言

如果把三者压成一套房间生产语言，我建议最后只保留下面 5 个核心问题。

每做一个房间，都必须先回答：

1. **它的房间职责是什么？**  
来源：`Minishoot`

2. **它在整张路网中是什么节点？**  
来源：`Silksong`

3. **它第一次进入和第二次回返，有没有语义变化？**  
来源：`Silksong + TUNIC`

4. **它让玩家更懂了什么？**  
来源：`TUNIC`

5. **它的奖励到底是资源，还是结构 / 解释 / 许可？**  
来源：三者综合

这 5 个问题一旦答清，房间通常就已经从“概念房”进入“可生产房”。

### 4.5 示巴星首个可玩切片的推荐架构模板

如果让我现在就给示巴星首切片画一个最推荐的骨架，我不会做成“连续大迷宫”，也不会做成“线性教程”。

我会做成：

**一个有主锚点、有两次认知抬升、有一次正式清算、有一次结构压缩的短环形切片。**

#### 推荐结构顺序

```text
R01 初入废墟引导房
 -> R02 开放压力房
 -> R03 测绘确认房
 -> R04 局部遮蔽/空间重读房
 -> R05 节点清算房
 -> R06 大钟门槛房
 -> R07 共鸣中继站房
 -> R08 钟城主锚点房（回环收束）
```

#### 这套顺序分别借了什么

`R01-R02`：

- 主要借 `Minishoot`
- 先把主方向、世界底噪和基本压力建立起来

`R03`：

- 主要借 `Silksong`
- 把“我在哪里”从模糊感受变成正式解释

`R04`：

- 主要借 `TUNIC`
- 用一次最便宜的空间重读，让示巴星开始出现“理解能改写路径”的味道

`R05`：

- 主要借 `Minishoot`
- 给玩家一次正式证明，确认战斗 loop 和关卡推进许可

`R06`：

- 借 `TUNIC + Silksong`
- 把章节门槛与长期承诺树起来，让玩家知道这里以后还会回来

`R07`：

- 主要借 `Silksong`
- 把前半段压力链压缩成未来捷径

`R08`：

- 借 `Silksong + Minishoot`
- 用主锚点收束，让玩家回到一个更可读、更可控的网络节点

### 4.6 如果只允许保留 3 条最关键的执行原则

如果最后只能记住 3 句话，我建议就是这 3 句：

#### 1. 先定义房间职责，再摆内容

这是 `Minishoot` 给 Ark 最值钱的纪律。

#### 2. 把捷径、地图、回返便利当成正式奖励

这是 `Silksong` 给 Ark 最值钱的提醒。

#### 3. 让理解先作为疑问出现，再作为通路被验证

这是 `TUNIC` 给 Ark 最值钱的翻译方式。

---

### 4.7 最后给示巴星的一句真正结论

示巴星首个可玩切片最不该做成的，是：

- 只有设定气质、没有房间秩序
- 只有探索面积、没有导航减负
- 只有谜语气氛、没有可验证的理解门槛

它最该做成的，是：

**玩家先被废墟吸引，随后在压力中学会识路，再通过一次认知抬升与一次结构压缩，明确感受到“这个世界不仅危险，而且可以被理解”。**

这才是三款参考游戏真正压到 `Project Ark` 身上之后，最值得交付的切片形态。
